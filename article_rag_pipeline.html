Мы строили Л.Е.С. — RAG-систему для технических консультаций по нормативной документации: своды правил, ГОСТы, постановления. Проектировщик задаёт вопрос, система находит нужные чанки из документов, LLM формулирует ответ.

Звучит просто. Но на вопросы вроде «Типы систем оповещения СОУЭ» или «Перечень разделов проектной документации по постановлению 87» система отвечала одним из двух:

🚫

NO_DATA

«В базе знаний нет точных данных по этому вопросу.» — при том что retrieval находил 24 релевантных чанка с quality: good.

❓

NEEDS_CLARIFICATION

«Запрос слишком широкий.» — до попытки поиска. «Виды работ по градкодексу» — вполне конкретный вопрос с конкретным ответом в базе.

✂️

Обрезанный ответ

«10 разделов, вот 2 из них...» — на вопрос о перечне из 10 пунктов. LLM физически не видела остальные 8.

⚠️

Ключевой инсайт: retrieval работал отлично. Чанки находились, качество было good, top_score 0.63. Проблема была в том, что происходило с ними после поиска.

Проблема #1

Удушение контекста:
конвейер-убийца

RAG-пайплайн — это не просто «найти и отдать». Между найденными чанками и текстом в промпте LLM стоит несколько слоёв обработки, каждый из которых обрезает данные.

Итог: LLM видела 4 чанка из 24 найденных. 83% информации терялось до генерации. LLM не могла перечислить все разделы — она просто их не видела.

Решение: динамическое масштабирование

Для структурированных и юридических запросов («перечислите», «какие разделы», «все типы») нужна другая стратегия — ширина, а не глубина.

proxy/routers/chat.py — динамические лимиты
is_structured = _is_structured_query(req.question)
- focus_max_chunks = env_int("RAG_CHAT_FOCUS_MAX_CHUNKS", 8)
- context_max_chunks = env_int("RAG_CONTEXT_MAX_CHUNKS", 6)
- context_chars = env_int("RAG_CHAT_CONTEXT_CHARS", 9000)
+ focus_max_chunks = 24 if is_structured else env_int("RAG_CHAT_FOCUS_MAX_CHUNKS", 8)
+ context_max_chunks = 24 if is_structured else env_int("RAG_CONTEXT_MAX_CHUNKS", 6)
+ context_chars = 32000 if is_structured else env_int("RAG_CHAT_CONTEXT_CHARS", 9000)
💡

Для 4B модели с контекстом 32k токенов передача 32k символов (~8k токенов) абсолютно безопасна. LLM не замедляется, но видит всю картину.

Проблема #2

Чанкинг — это не
«порезать файл на куски»

Большинство туториалов по RAG показывают чанкинг как: взять PDF → разбить на 512-токенные блоки → загрузить в векторную БД. Это работает на демо. В продакшене — нет.

🏗️

Структура документа — это данные

Нормативный документ — иерархия: раздел → пункт → подпункт. Если разрезать посередине абзаца, смысл теряется. Чанкер должен понимать структуру.

📏

Размер чанка — это компромисс

Маленький чанк — точный поиск, но нет контекста. Большой чанк — контекст есть, но вектор «размывается» и поиск хуже. Наш выбор: ~1000 символов + sliding window при подаче в LLM.

🔗

Чанки — связная сеть

Каждый чанк знает своих соседей (parent_id, chunk_ord). При генерации мы расширяем контекст: берём чанк ± radius соседей. Для списочных запросов — radius=0, максимальный охват.

🏷️

Метаданные важнее вектора

doc_name, domain, section_heading, page — это фильтры. Без них hybrid search смешивает СОУЭ с пожарными спринклерами просто потому что векторы близко.

🔍

Реальный кейс: вопрос «Расстояние между пожарными извещателями» попадал в dataset NTD_FIRE и находил чанки о спринклерных оросителях — тоже пожарные, тоже расстояния, но совершенно другой документ (СП 5.13130 vs СП 10.13130). Без правильного routing по домену и метаданным — это несортируемый шум.

Hybrid retrieval: почему BM25 + вектор лучше каждого по отдельности

Векторный поиск плохо работает с терминами-аббревиатурами: «СОУЭ», «АУПС», «ГрК РФ» — модель embedding видит их как редкие токены, семантика размыта. BM25 (лексический поиск) находит их точно. Объединение через RRF (Reciprocal Rank Fusion) даёт лучшее из двух миров.

Проблема #3

Валидатор-параноик:
когда безопасность убивает пользу

Т.О.С.К.А. (Технологический Обеспечитель Семантической Корректности Ответов) — наш слой валидации. Идея правильная: перед тем как отдать ответ пользователю, проверить что LLM не выдумала факты. Реализация была катастрофической.

Старый rules-валидатор: literal substring match
Контекст:
«...ширина эвакуационных выходов должна быть не менее 1,2 м при числе эвакуирующихся групп М2-М4 более 15 человек...»
Ответ LLM:
«Минимальная ширина эвакуационных выходов предусматривается не менее 1,2 м в следующих случаях: при числе эвакуирующихся МГН более 15 человек...»
Проверка: ответ входит в контекст как подстрока?
✗ NO_DATA — не совпадает дословно

LLM никогда не копирует текст дословно — она перефразирует, структурирует, добавляет контекст. Literal match гарантированно возвращал NO_DATA на любой нетривиальный ответ. Это детерминированная блокировка, замаскированная под проверку качества.

Решение: лексическое перекрытие

Вместо «входит ли ответ в контекст» спрашиваем: «насколько значимые слова из ответа присутствуют в контексте?» Если ≥35% ключевых токенов ответа есть в контексте — ответ подтверждён.

overlap = hits / meaningful_tokens
где hits = | {t ∈ answer_tokens : t ∈ context} |
VERIFIED if overlap ≥ 0.35 AND numbers_ok
Числовые заявления проверяются отдельно: если ответ содержит число (1,2 м, 15 чел), оно должно присутствовать в контексте.
Новый rules-валидатор: lexical overlap
Токены:
минимальная ширина эвакуационных выходов предусматривается менее 1,2 следующих случаях числе эвакуирующихся
Лексическое перекрытие: 89%
✓ VERIFIED — lexical_overlap: 0.89

Выводы

Что мы узнали
(чтобы ты не узнавал снова)

🔬

Retrieval ≠ RAG

Хороший retrieval — необходимое, но не достаточное условие. Валидируй каждый этап: concentrate → expand → build_context → LLM prompt. Логируй сколько чанков доходит до LLM.

🧪

Тестируй валидатор отдельно

Вызывай /api/validate напрямую с реальными парами (вопрос, ответ, контекст). Literal match никогда не должен быть production-стратегией для нейросетевых ответов.

💾

Проверяй векторный индекс

После индексации: SELECT COUNT(*) WHERE status='ERROR'. Не доверяй state-файлам индексатора. Сверяй кол-во чанков в Qdrant с ожидаемым по SQLite.

📐

Масштабируй контекст под запрос

«Перечисли все разделы» и «что такое огнестойкость» — разные задачи. Первому нужна ширина (24 чанка, 32k символов), второму — глубина (6 чанков с соседями).

⚡

Размер модели ≠ качество ответа

В RAG качество определяется контекстом. 4B модель с правильным контекстом бьёт 9B модель с обрезанным. И работает в 7× быстрее на ограниченной RAM.

🗺️

Routing — это архитектура

Не смешивай разные домены в одном поиске без фильтрации. СОУЭ и спринклеры — оба «пожарные», но это разные документы, разные вопросы, разный контекст.

✅
