# Как мы защитили RAG от самого себя: отравление чанками, петля SafeRAG и судья на 4B

> Практическая заметка из боевой разработки Л.Е.С. — локального инженерного RAG-стека на Apple Silicon.

---

## Проблема: ответ ни о чём

Пользователь спрашивает: *«Каковы требования к несущей способности буроинъекционных свай?»*

Система находит 5 релевантных чанков в Qdrant. Беда в том, что из 5 чанков — 3 реально про сваи, а 2 — про гидротехнические сооружения, которые по векторной близости оказались рядом. LLM послушно читает весь контекст и пишет ответ, в котором смешаны СП про сваи и СНиП про плотины. Формально не галлюцинация — всё из документов. Но пользователю бесполезно.

Это **контаминация контекста** — классическая болезнь наивного RAG, где top-k это просто ближайшие соседи без учёта того, что они из разных документов на разные темы.

## Слой 1: Концентрация источников

Первый рубеж — до LLM, чисто алгоритмический.

После retrieval и опционального реранкинга запускается `_concentrate_sources()`:

```python
def _concentrate_sources(chunks, max_docs=2, min_score=0.45):
    # 1. Отбрасываем откровенно слабые чанки
    filtered = [c for c in chunks if c.score >= min_score]
    if not filtered:
        best = max(c.score for c in chunks)
        filtered = [c for c in chunks if c.score >= best * 0.8]

    # 2. По каждому документу берём его максимальный score
    doc_max = {}
    for c in filtered:
        doc_max[c.doc_name] = max(doc_max.get(c.doc_name, 0), c.score)

    # 3. Оставляем только top-2 документа
    top_docs = set(sorted(doc_max, key=lambda d: -doc_max[d])[:max_docs])
    return [c for c in filtered if c.doc_name in top_docs]
```

**Что это даёт:** если Qdrant вернул чанки из 4 документов, в контекст LLM попадут только 2 наиболее релевантных. Лишние источники отрезаются до того, как LLM их увидит.

В логах это выглядит так:
```
[FOCUS] Отсечено 1 нерелевантных источников: {'ГОСТ 34259-2017. Грунты. Метод.docx'}
[FOCUS] После концентрации: 3 чанков из 2 источников
```

**Почему не просто увеличить порог similarity?** Потому что абсолютные значения score зависят от модели и датасета. Документ про плотины может иметь score 0.62, а лучший чанк про сваи — 0.71. Порог 0.65 срежет плотины, но при другом вопросе срежет и нужный документ. Концентрация по top-N документам более адаптивна.

## Слой 2: Судья (Т.О.С.К.А.)

После того как LLM сгенерировал ответ, его проверяет отдельная модель — **Qwen3-4B-Instruct** (val_engine), работающая параллельно с основной Qwen3.5-9B на том же Apple Silicon.

Судья получает тройку: вопрос + контекст + ответ, и выносит вердикт одним словом.

```python
# mlx_host.py — /api/validate
prompt = val_engine.apply_chat_template([{
    "role": "system",
    "content": "Ты — строгий валидатор. Отвечай ТОЛЬКО одним словом: VERIFIED, NO_DATA или HALLUCINATION."
}, {
    "role": "user",
    "content": (
        f"Вопрос: {req.question}\n"
        f"Контекст: {req.context[:1500]}\n"
        f"Ответ для проверки: {req.answer[:1000]}\n\n"
        "VERIFIED — ответ подтверждается контекстом И отвечает на заданный вопрос.\n"
        "NO_DATA — контекст не содержит нужных данных.\n"
        "HALLUCINATION — ответ противоречит контексту, содержит выдумки, "
        "или технически верен но НЕ отвечает на заданный вопрос.\n"
        "Одно слово:"
    )
}], enable_thinking=False)  # <-- критично: без <think> блока, иначе таймаут
```

Ключевое — `enable_thinking=False`. Qwen3 умеет думать перед ответом (режим Chain-of-Thought), но валидатору не нужны рассуждения — нужен мгновенный вердикт. Без этого флага судья думает 30–60 секунд и выдаёт `UNKNOWN`.

### Три вердикта

| Статус | Значение | Действие |
|--------|----------|----------|
| `VERIFIED` | Ответ подтверждён и по теме | Отдаём пользователю |
| `NO_DATA` | Документов по теме нет | Сообщение «нет данных» |
| `HALLUCINATION` | Противоречие или не по теме | SafeRAG retry / блокировка |

Важное расширение: HALLUCINATION теперь включает случай когда ответ **технически верен, но не отвечает на вопрос**. Это закрывает дыру — раньше контаминированный ответ мог пройти валидацию («всё из документов, ничего не придумано»), даже если был совершенно не о том.

### Как живёт val_engine

Модель загружается lazy — только при первом запросе на валидацию — и выгружается через 120 секунд простоя (`TTL=120`). Основная модель (9B) держится 300 секунд.

При нехватке памяти (≥70% swap) `memory_guard_loop()` принудительно выгружает val_engine первой — она менее критична.

## Слой 3: SafeRAG петля

Концентрация и судья дополняются **петлёй повторных попыток**:

```
Попытка 1
  Контекст: top-2 документа, 12 000 символов
  Промпт: нормальный («отвечай только по контексту»)
  → генерация → судья
    VERIFIED / NO_DATA → выход ✓
    HALLUCINATION      → Попытка 2

Попытка 2
  Контекст: top-1 документ, 6 000 символов
  Промпт: строгий («дословно, без обобщений»)
  → генерация → судья
    VERIFIED / NO_DATA → выход ✓
    HALLUCINATION      → SAFE_FALLBACK (блокировка)
```

Строгий промпт для попытки 2:

```
Ты — строгий технический консультант.
Отвечай ТОЛЬКО тем, что явно написано в контексте — дословно, без домыслов и обобщений.
Если точного ответа нет — напиши: 'В базе знаний нет точных данных по этому вопросу.'
```

Если обе попытки дали HALLUCINATION, пользователь получает:

```
⚠ Система безопасности (Т.О.С.К.А.) не смогла подтвердить ответ из базы знаний.
Попробуйте переформулировать вопрос или выбрать другой датасет.
```

### Почему именно 2 попытки, не 3?

- 1 попытка = оптимистично (не обнаружим проблему при тонкой контаминации)
- 2 попытки = разумный компромисс: +50% латентности в плохом случае, но без тройного штрафа
- 3 попытки = 3× время генерации, пользователь ждёт 3–5 минут

Вторая попытка принципиально другая по структуре (не повтор), поэтому имеет смысл.

## Архитектура в целом

```
Запрос пользователя
       │
       ▼
  Qdrant retrieve (top-8)
       │
       ▼
  Реранкер (Qwen3-4B, batch) → top-5     [опционально]
       │
       ▼
  _concentrate_sources()     → top-2 docs  ◄── Слой 1
       │
  ┌────┴────────────────────────────────────────┐
  │  SafeRAG Loop (до 2 итераций)               │
  │                                              │
  │  Попытка 1: нормальный промпт + 12k контекст │
  │  Попытка 2: строгий промпт + 6k + top-1 doc │
  │                                              │
  │  Каждая итерация:                            │
  │    Qwen3.5-9B → генерация                   │
  │    Qwen3-4B   → /api/validate  ◄── Слой 2   │
  │                                              │
  └─────────────────────────────────────────────┘
       │
       ▼
  VERIFIED → ответ пользователю
  NO_DATA  → "нет данных в базе"
  HALLUCINATION (2 попытки) → SAFE_FALLBACK  ◄── Слой 3
```

## Что это даёт на практике

В логах реального запроса:

```
[FOCUS] Отсечено 1 нерелевантных источников: {'ГОСТ 34259-2017. Грунты. Метод.docx'}
[FOCUS] После концентрации: 4 чанков из 2 источников
[CHAT] attempt=1 model=Qwen3.5-9B tokens=143
[TOSKA] attempt=1 → VERIFIED
```

**Латентность:**
- Нормальный путь (attempt=1, VERIFIED): +3–5 сек на судью → итого 25–35 сек
- Retry (attempt=2): +ещё 25–35 сек → итого 50–70 сек
- Блокировка: суммарно аналогично retry, но пользователь получает предупреждение

**Ресурсы:**
- val_engine (Qwen3-4B) живёт только 120 сек после последнего вызова
- RAM: ~2.5 GB при загрузке, 0 в выгруженном состоянии
- При retry val_engine уже загружен → нет overhead на инициализацию
