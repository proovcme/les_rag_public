# MVP Roadmap

## Цель

Собрать первую проверяемую версию АРТЕЛЬ, которая закрывает полный цикл:

`задание -> AI-спецификация -> Revit-плагин -> проверка -> отправка RFA -> публикация в каталог -> поиск и повторное использование`.

## Базовые допущения

- MVP делается для одной компании без multi-tenant логики.
- В MVP плагин работает только с документами Family Editor.
- Геометрия создается вручную разработчиком семейства.
- Автоматизация Revit ограничена параметрами, типами, значениями и проверками.
- ФОП/shared parameters загружается как TXT-файл и парсится backend-сервисом.
- AI-спецификация всегда требует подтверждения человеком перед выдачей в Revit.

## Фаза 0. Подготовка

Срок: 3-5 рабочих дней.

Результат:

- утвержденный MVP scope;
- список поддерживаемых версий Revit;
- выбранный backend/frontend stack;
- выбранный способ авторизации плагина;
- согласованный формат спецификации семейства.

Критерий готовности:

- можно начинать разработку без открытых архитектурных блокеров.

## Фаза 1. Web Core

Срок: 2 недели.

Состав:

- список заданий;
- создание задания;
- карточка задания;
- загрузка исходников;
- хранение статусов;
- базовые роли `bim_manager` и `family_developer`;
- карточка спецификации без AI;
- ручное редактирование параметров и типов.
- базовый каталог как отдельный web workspace: список, карточка, версии, скачивание.

Критерий готовности:

- BIM-менеджер может создать задание, приложить файлы, вручную собрать спецификацию для плагина и видеть принятые семейства в каталоге.

## Фаза 2. FOP и AI-разбор

Срок: 2 недели.

Состав:

- загрузка TXT shared parameters;
- парсинг групп, GUID, имен и типов параметров;
- загрузка PDF/DOCX/XLSX исходников;
- извлечение черновика спецификации;
- сопоставление требований с ФОП;
- список конфликтов и недостающих параметров;
- подтверждение спецификации человеком.

Критерий готовности:

- из пакета задания можно получить редактируемую спецификацию с параметрами, типами и чеклистом.

## Фаза 3. Revit Add-In MVP

Срок: 3 недели.

Состав:

- авторизация;
- список назначенных заданий;
- панель задания;
- получение спецификации;
- добавление shared/family parameters;
- создание family types;
- установка значений параметров типов;
- локальная проверка активного family-документа;
- отправка RFA и отчета проверки.

Критерий готовности:

- разработчик может открыть задание в Revit, применить параметры/типы, проверить семейство и отправить результат.

## Фаза 4. Review и Catalog

Срок: 2 недели.

Состав:

- просмотр отправленной версии;
- просмотр validation report;
- принятие/отклонение версии;
- публикация принятой версии в каталог;
- карточка каталога;
- скачивание актуальной RFA;
- поиск по имени, категории и тегам.

Критерий готовности:

- принятая версия семейства появляется в каталоге и доступна для скачивания.

## Фаза 5. Hardening

Срок: 1-2 недели.

Состав:

- обработка ошибок загрузки файлов;
- ограничения типов файлов;
- audit log ключевых действий;
- smoke tests backend;
- ручной QA Revit-плагина;
- проверка GitHub Pages/документации;
- подготовка demo-script.

Критерий готовности:

- MVP можно показать BIM-команде на 2-3 реальных заданиях.

## Приоритет реализации

1. Ручная спецификация без AI.
2. Парсинг ФОП.
3. Revit-плагин для параметров и типов.
4. Проверка семейства.
5. AI-разбор задания.
6. Приемка и каталог.

Причина: Revit-плагин и формат спецификации являются системным ядром. AI без надежного action/data contract быстро превращается в демонстрацию без эксплуатационной ценности.

## MVP Definition of Done

- Полный workflow проходит на одном тестовом семействе.
- Все действия плагина логируются в отчете.
- AI-спецификация не выдается в Revit без подтверждения.
- Принятый RFA доступен из каталога.
- Документация содержит актуальный workflow, API contract и ограничения MVP.
