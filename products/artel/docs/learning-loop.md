# Learning Loop

## Принцип

АРТЕЛЬ должен становиться лучше по мере разработки и приемки новых семейств.

Но само количество RFA-файлов не улучшает систему автоматически. Ценность появляется только если каждое семейство сохраняется вместе с контекстом разработки, проверкой и результатом приемки.

Ключевая обучающая единица:

```text
задание -> исходники -> AI-спецификация -> правки BIM-менеджера -> действия в Revit -> RFA -> проверка -> приемка -> использование в проектах
```

## Что нужно сохранять

Для каждого принятого семейства система должна хранить:

- исходное задание;
- приложенные исходники;
- использованный ФОП/shared parameter profile;
- использованный шаблон RFT/RFA;
- AI-черновик спецификации;
- финальную утвержденную спецификацию;
- изменения, внесенные человеком;
- список действий Revit-плагина;
- итоговый RFA;
- validation report;
- замечания приемки;
- статус публикации в каталоге;
- историю версий;
- факты использования: скачивания, проекты, повторные обновления.

## Текущий API-контур

Backend MVP уже умеет принимать validation report от Revit add-in:

```http
POST /api/revit/tasks/{taskId}/validation-reports
GET /api/validation-reports?taskId={taskId}
```

И отдавать LES-ready learning case:

```http
GET /api/tasks/{taskId}/learning-case
GET /api/validation-reports/{reportId}/learning-case
```

Операторский путь от JSON-отчета Revit add-in до LES:

```bash
python3 tools/ingest_artel_validation_report.py \
  --report 'local_private_archive/artel_validation_reports/validation_*.json' \
  --artel-url http://127.0.0.1:5057 \
  --task-id task_0241 \
  --runtime-root /Users/ovc/Projects/LES_v2_reinstall_stress \
  --proxy-url http://127.0.0.1:8050 \
  --verify-search
```

Скрипт нормализует report, делает `POST /api/revit/tasks/{taskId}/validation-reports`,
забирает `GET /api/validation-reports/{reportId}/learning-case`, пишет markdown
projection в `RAG_Content/ARTEL/family_learning_cases/` и запускает LES sync.
Backend сохраняет reports в `ARTEL_DATA_DIR/validation_reports/` и поднимает
их обратно после restart, чтобы learning loop не терял историю приемки.

Чтобы посадить в LES все накопленные backend reports:

```bash
python3 tools/seed_artel_backend_reports.py \
  --artel-url http://127.0.0.1:5057 \
  --task-id task_0241 \
  --runtime-root /Users/ovc/Projects/LES_v2_reinstall_stress \
  --proxy-url http://127.0.0.1:8050 \
  --verify-search
```

Это закрывает минимальную петлю:

```text
Revit family validation -> ARTEL report -> LES LEARNING_CASE -> retrieval
```

## Как система улучшается

### 1. Reference Library

Принятые семейства становятся базой эталонных образцов.

Новый запрос можно сравнивать с уже принятыми семействами и предлагать:

- похожие семейства;
- подходящий шаблон;
- типовой набор параметров;
- типовую структуру типоразмеров;
- известные риски.

### 2. Archetypes

Система группирует семейства по архетипам:

- шкаф;
- дверь;
- светильник;
- диффузор;
- оборудование;
- стеллаж;
- сантехнический прибор;
- generic equipment.

Архетип помогает выбрать не только похожее семейство, но и подходящий способ разработки.

### 3. Recipes

Для повторяемых архетипов формируются recipes:

```text
cabinet_box:
- reference planes: Left, Right, Front, Back, Top, Bottom
- parameters: Width, Height, Depth
- materials: BodyMaterial, FrontMaterial
- types: from XLSX
- checks: required parameters, type values, material assignment
```

Recipe — это не AI-текст, а структурированный план, который Revit-плагин может частично выполнить и проверить.

### 4. Retrieval

При новом задании система ищет похожие принятые семейства:

- по категории;
- по параметрам;
- по типоразмерам;
- по материалам;
- по тексту задания;
- по архетипу;
- по производителю или серии.

Результат: AI не начинает с пустого листа, а работает от проверенного образца.

### 5. Adaptation

AI предлагает не произвольное решение, а план адаптации:

```text
Взять CAT-0148 "Шкаф архивный металлический" как образец.
Заменить типоразмеры по новой XLSX.
Добавить параметр ADSK_Серия.
Обновить материалы фасада.
Сохранить recipe cabinet_box.
Запустить проверки параметров и типов.
```

### 6. Validation Memory

Система запоминает ошибки, которые раньше находили в похожих семействах:

- не заполнены обязательные ADSK-параметры;
- конфликт единиц;
- неверная категория;
- дубли параметров;
- тяжелая геометрия;
- отсутствующие материалы;
- неполные типы.

При новом задании эти ошибки заранее попадают в checklist.

## Что не является обучением

Не считается достаточным:

- просто загрузить RFA в каталог;
- просто сохранить текст задания;
- просто получить AI-ответ;
- просто скачать семейство из внешней библиотеки.

Нужна полная связка входа, решения, результата и проверки.

## Fine-Tuning не является первым шагом

На раннем этапе лучше не начинать с обучения модели.

Более практичный порядок:

1. Накопить структурированные кейсы.
2. Настроить поиск похожих семейств.
3. Формировать архетипы и recipes.
4. Использовать retrieval + AI для адаптации.
5. Только после накопления достаточного объема качественных кейсов рассматривать fine-tuning или специализированные модели.

Причина: большая часть ценности сначала появляется не от новой модели, а от качественной базы примеров, правил и проверок.

## Метрики улучшения

Система действительно становится лучше, если со временем:

- AI-спецификация требует меньше ручных правок;
- чаще находится подходящее эталонное семейство;
- уменьшается число ошибок на приемке;
- быстрее создаются типы и параметры;
- растет доля семейст, прошедших проверку с первого раза;
- увеличивается повторное использование каталога;
- меньше заданий блокируется из-за недостатка данных.

## Продуктовый вывод

Каждое принятое семейство должно пополнять не только каталог скачивания, но и базу разработки.

Ценность АРТЕЛЬ растет не от количества файлов, а от качества накопленной связки:

```text
задание -> спецификация -> действия -> RFA -> проверка -> приемка -> использование
```

Именно эта связка позволяет делать следующие семейства быстрее, точнее и с меньшим количеством ошибок.
