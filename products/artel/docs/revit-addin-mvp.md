# Revit Add-In MVP

## Назначение

Revit-плагин в MVP является исполнительной средой для утвержденной спецификации семейства.

Он не должен решать задачу разработки семейства целиком. Его задача — снять ручную рутину и дать проверяемый канал между веб-сервисом и Family Editor.

## Поддержка Revit

Требует уточнения перед реализацией.

Предварительная рекомендация:

- Revit 2023-2025;
- .NET Framework 4.8 для совместимости;
- C# + WPF panel;
- работа только в Family Editor на первом этапе.

## Основные команды

### Login

Авторизация пользователя и сохранение access token.

Минимальные требования:

- ввод backend URL;
- ввод token или OAuth/device flow;
- проверка текущего пользователя;
- выход из аккаунта.

### My Tasks

Показ назначенных заданий.

Данные:

- номер;
- название;
- категория;
- статус;
- срок;
- readiness/validation status.

### Task Package

Получение утвержденной спецификации и исходников.

Плагин должен явно показывать:

- семейство;
- категорию;
- список параметров;
- типы;
- чеклист;
- файлы задания.

### Apply Parameters

Добавление параметров в активный family document.

Правила:

- не добавлять дубликаты;
- shared parameters добавлять по GUID;
- family parameters добавлять только из approved specification;
- перед изменением показывать список действий;
- после выполнения сохранить result log.

### Create Types

Создание типов семейства.

Правила:

- отсутствующие типы создаются;
- существующие типы не перезаписываются без подтверждения;
- значения записываются только для существующих параметров;
- ошибки записи не прерывают весь batch, но попадают в report.

### Validate Family

Проверка активного family document.

Минимальные проверки:

- документ открыт в Family Editor;
- категория соответствует спецификации;
- обязательные параметры существуют;
- типы из спецификации существуют;
- обязательные значения заполнены;
- формулы параметров читаются без ошибок;
- файл можно сохранить.

### Submit

Отправка RFA и отчета в веб-сервис.

Правила:

- submit доступен только после validation run;
- при `fail` нужен явный override/comment или submit запрещается;
- отправляется текущая версия RFA;
- статус задания меняется на `submitted_for_review`.

## UI-панель

Минимальная структура:

- верхняя зона: пользователь, backend, обновить;
- список заданий;
- карточка выбранного задания;
- вкладки `Спецификация`, `Параметры`, `Типы`, `Проверка`, `Отправка`;
- нижняя зона действий.

## Action Preview

Перед изменениями в Revit плагин показывает preview:

```json
{
  "action": "apply_parameters",
  "willCreate": ["ADSK_Наименование", "ADSK_Код изделия", "Ширина"],
  "willSkip": ["Высота"],
  "warnings": ["Параметр Материал фасада отсутствует в ФОП"]
}
```

## Action Log

Каждое действие плагина пишет локальный и отправляемый лог:

```json
{
  "timestamp": "2026-06-06T10:00:00Z",
  "action": "create_type",
  "target": "Шкаф 800x400x1800",
  "status": "success",
  "message": null
}
```

## Ограничения MVP

- Плагин не строит геометрию.
- Плагин не создает reference planes.
- Плагин не редактирует constraints.
- Плагин не меняет вложенные семейства.
- Плагин не работает с проектными RVT-документами.
- Плагин не вызывает AI напрямую.

## Технические риски

### Shared Parameters

Добавление shared parameters требует корректного файла ФОП и GUID. Нужно заранее определить, где плагин получает физический shared parameter file: скачивает из backend или использует локальный temporary file.

### Units

Значения length/number должны передаваться в нейтральном формате и конвертироваться в internal Revit units.

### Category

Не все категории одинаково доступны в Family Editor. Проверку категории нужно делать до выполнения batch-действий.

### Version Compatibility

API Revit отличается по версиям. MVP должен либо ограничиться одной версией Revit, либо иметь отдельные build targets.
