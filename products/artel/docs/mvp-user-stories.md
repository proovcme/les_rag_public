# MVP User Stories

## Роли

### BIM Manager

Отвечает за постановку задания, стандарты, приемку и публикацию семейства.

### Family Developer

Разрабатывает семейство в Revit, применяет спецификацию и отправляет результат.

### Catalog User

Ищет и скачивает утвержденные семейства.

## Epic 1. Задания

### US-001. Создать задание

Как BIM-менеджер, я хочу создать задание на разработку семейства, чтобы зафиксировать цель, категорию, срок и исполнителя.

Acceptance criteria:

- можно указать название, описание, категорию Revit, срок и исполнителя;
- новое задание получает номер вида `FAM-0001`;
- статус нового задания — `draft`;
- действие попадает в audit log.

### US-002. Загрузить исходники

Как BIM-менеджер, я хочу приложить к заданию ТЗ, ФОП, шаблон и таблицу типоразмеров, чтобы разработчик и AI работали с одним пакетом данных.

Acceptance criteria:

- поддерживаются PDF, DOCX, XLSX, TXT, RFT, RFA, PNG, JPG, DWG;
- каждый файл виден в карточке задания;
- для файла сохраняются имя, тип, размер, автор и дата загрузки;
- неподдерживаемые типы отклоняются.

### US-003. Перевести задание на AI-разбор

Как BIM-менеджер, я хочу отправить пакет задания на AI-разбор, чтобы получить черновик спецификации.

Acceptance criteria:

- команда доступна только при наличии исходников;
- статус меняется на `ready_for_ai_analysis`;
- после завершения анализа появляется черновик спецификации;
- при ошибке анализа сохраняется диагностическое сообщение.

## Epic 2. Спецификация

### US-004. Проверить AI-спецификацию

Как BIM-менеджер, я хочу проверить AI-спецификацию перед выдачей в Revit, чтобы исключить ошибочные автоматические действия.

Acceptance criteria:

- спецификация открывается в редактируемом виде;
- видны параметры, типы, материалы и чеклист;
- видны конфликты ФОП;
- спецификацию можно утвердить или вернуть на доработку.

### US-005. Редактировать параметры

Как BIM-менеджер, я хочу вручную исправить параметры спецификации, чтобы привести ее к корпоративному стандарту.

Acceptance criteria:

- можно добавить, удалить и изменить параметр;
- для shared parameter выбирается GUID из ФОП;
- обязательность, instance/type и группа параметра редактируются;
- изменения версионируются.

### US-006. Редактировать типы

Как BIM-менеджер, я хочу редактировать типы семейства, чтобы выдать разработчику точную таблицу типоразмеров.

Acceptance criteria:

- можно добавить, удалить и переименовать тип;
- можно задать значения параметров по типам;
- значения валидируются по типам данных;
- пустые обязательные значения подсвечиваются.

## Epic 3. Revit-плагин

### US-007. Получить список заданий

Как разработчик, я хочу видеть назначенные мне задания внутри Revit, чтобы не искать их во внешних каналах.

Acceptance criteria:

- плагин показывает только доступные пользователю задания;
- видны номер, название, категория, статус и срок;
- можно открыть карточку задания;
- список обновляется по команде пользователя.

### US-008. Применить параметры

Как разработчик, я хочу применить обязательные параметры из спецификации, чтобы не добавлять их вручную.

Acceptance criteria:

- shared parameters добавляются по GUID из ФОП;
- family parameters добавляются по описанию спецификации;
- существующие параметры не дублируются;
- результат действия отображается в отчете.

### US-009. Создать типы

Как разработчик, я хочу создать типы из спецификации, чтобы ускорить подготовку типоразмеров.

Acceptance criteria:

- плагин создает отсутствующие типы;
- существующие типы можно пропустить или обновить;
- значения параметров записываются по типам;
- ошибки записи попадают в отчет.

### US-010. Проверить семейство

Как разработчик, я хочу проверить открытое семейство по спецификации, чтобы отправить на приемку только корректный RFA.

Acceptance criteria:

- проверяется категория;
- проверяются обязательные параметры;
- проверяются типы;
- проверяются пустые обязательные значения;
- результат сохраняется как validation report.

### US-011. Отправить результат

Как разработчик, я хочу отправить RFA и отчет обратно в веб-сервис, чтобы BIM-менеджер мог принять работу.

Acceptance criteria:

- отправляется текущий RFA;
- отправляется validation report;
- статус задания меняется на `submitted_for_review`;
- версия получает номер и автора.

## Epic 4. Приемка и каталог

### US-012. Принять версию

Как BIM-менеджер, я хочу принять отправленную версию, чтобы зафиксировать ее как пригодную к использованию.

Acceptance criteria:

- видны RFA, отчет проверки и комментарии разработчика;
- версию можно принять или отклонить;
- при отклонении требуется комментарий;
- решение попадает в audit log.

### US-013. Опубликовать в каталог

Как BIM-менеджер, я хочу опубликовать принятую версию в каталог, чтобы проектные команды могли ее найти.

Acceptance criteria:

- создается или обновляется catalog item;
- текущая версия становится active;
- карточка содержит название, категорию, описание, теги и ссылку на RFA;
- карточка доступна в поиске.

### US-014. Найти семейство

Как пользователь каталога, я хочу найти утвержденное семейство, чтобы скачать актуальную версию.

Acceptance criteria:

- поиск работает по имени, категории и тегам;
- карточка показывает актуальную версию;
- можно скачать RFA;
- устаревшие версии не показываются как актуальные.

## MVP Non-Functional Requirements

- Все изменения статусов логируются.
- Все файлы имеют владельца и дату загрузки.
- Плагин не выполняет AI-действия без подтвержденной спецификации.
- Backend API возвращает стабильные machine-readable ошибки.
- Валидация Revit-плагина должна работать без доступа к AI.
