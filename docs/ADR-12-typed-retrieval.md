# ADR-12 — Типизированный многоуровневый ретрив + формирование ответа по интенту

Статус: **предложено** (2026-06-17, дизайн с оператором). Дополняет [ADR-1](LES3_PLAN.md) (эволюция, не переписывание), [ADR-3](LES3_PLAN.md) (cross-encoder реранк), [ADR-11](LES3_PLAN.md) (LLM-минимализм). Связано с графом знаний (W17, `PROJECT_KNOWLEDGE_GRAPH.md`).

## Контекст — два недостающих слоя (а не баги)

Текущий чат: классификатор выбирает **датасет** (NTD_FIRE) → `retrieve_chat_chunks` делает **плоский гибридный поиск (dense+sparse+RRF+rerank) сразу по всем чанкам всех документов датасета**. Это «гугл по чанкам»: документного уровня нет, форму ответа никто не держит.

Доказательство (кейс «требования к серверным», 2026-06-17):
- **Вход:** ретрив тянет смежные СП (здания судов/прокуратур — там слово «серверная» буквально), а ядро — СП 485 (газовое ПТ), СП 486 (перечень помещений под АУПТ/АУПС, **таблица-приложение**), СП 484 (АУПС) — не поднимается даже точечными запросами (`NO_DATA`). Ответ в таблицах приложений, плоский чанк-поиск их не достаёт.
- **Выход:** на `gpt-4o-mini` ответ куцый и поверхностный; на `gpt-4.1` — «полотно» (эссе с заголовками и закрывашкой). Объём = чистый дефолт модели; контракта формы нет, стили («Краткие тезисы») живут только в GUI и опциональны.

Вывод: системе не хватает **двух слоёв** — на входе «куда искать» (документный/проектный уровень) и на выходе «как отвечать» (форма по интенту). Между ними — готовое ядро (гибрид, реранк, валидатор, каталог датасетов).

## Решение

### Слой 1 — типизированный многоуровневый ретрив (класс → узел → лист)

Поиск становится **спуском по типизированному дереву**, а фрагменты ищутся только в выбранных узлах.

**Верхний уровень — класс источника** (обязательный, первый):

| Класс | Что | Текущий датасет |
|---|---|---|
| норматив | СП / ГОСТ / СНиП | `NTD_*` (домен — фасет внутри) |
| закон/кодекс | кодексы, законы | `GKRF_Index` |
| проект | РД/ПД, чертежи, BIM-модели | `CAD_BIM_Index` + файлы объекта |
| письмо | переписка | `MAIL_Index` |
| договор | контракты | (новый) |
| смета | сметы/расценки | `TABLE_SMETA_Index` |
| документ/прочее | книги, справочники | `BOOKS_Index`, `exports_Index`, `ARTEL_Index` |

**У каждого класса своя глубина** (pluggable — не общий шаблон):
- **норматив:** класс → **документ** (по каталогу: шифр + название + область применения + оглавление + таблицы-приложения) → раздел/пункт → фрагмент. Фасет: домен (пожарная/электрика/…).
- **закон:** класс → кодекс/закон → статья → пункт.
- **проект:** класс → **две независимые оси-фасета: [объект] × [раздел РД]** (АР/КР/ОВ/ВК/ЭОМ/ПОС/…) → лист/документ → фрагмент. Поверх W17 `les_projects` + справочник разделов.
- **письмо:** класс → тред/контрагент → письмо → фрагмент.
- **договор:** класс → контрагент/объект → договор → пункт.
- **смета:** класс → смета/объект → раздел сметы → позиция.
- **документ/прочее:** класс → документ → фрагмент (плоско).

**Поток запроса (3 стадии):**
- **Стадия 0 — класс + интент.** Детерминированный классификатор (KOT-словарь + сигналы `doc_type`/`domain`) определяет класс(ы) и **форму ответа** (см. слой 2). Может вернуть несколько классов.
- **Стадия 1 — узлы (LLM-роутер по каталогу + кэш).** Внутри класса — компактный **каталог документов** (`шифр + название + «область применения»`, 1 строка/документ), и **LLM выбирает регулирующие документы** по вопросу. Решения кэшируются (запрос→документы) → карта строит себя из использования, доля LLM-вызовов → 0 (ADR-11 «пиши в кэш»). Реализация: `proxy/services/doc_router.py`, за флагом `LES_TYPED_RETRIEVAL`.

  **Почему LLM, а не поиск (эмпирика 2026-06-17):** поверхностный роутинг — лексика, sparse BM25 (sum и max), косинус по scope-карточке — **все falsified** на кейсе «серверные»: слово «серверная» буквально сидит в СП типов зданий (тюрьмы 509/суды/прокуратуры — там это комната в перечне), а управляющие нормы (СП 486/485) описывают её **категорией** («помещения с ЭВМ»), без слова. Вокабулярный разрыв surface-поиск не мостит — это работа LLM (ADR-11: «понимание, мост»). Ручная карта категория→нормы отвергнута как немасштабируемая (вопрос оператора «это системно?»). LLM-роутер по каталогу — общий механизм для всех классов, self-improving через кэш. Анти-галлюцинация: возвращённые шифры валидируются по каталогу. Проверено вживую: «серверные» → СП 2/6/7/484/485, ответ с п. 6.6.21/6.6.22 СП 484 (раньше плоский поиск давал СП судов/прокуратур).
- **Стадия 2 — листья.** Текущий гибрид dense+sparse+rerank, но **с фильтром по выбранным узлам** (Qdrant payload filter по `doc_id`/узлу), а не по всему датасету.
- Слияние мультикласс + формирование ответа.

**Каталог строится детерминированно** (ADR-11, минимум ЛЛМ): шифр — regex из имени; название и **«Область применения»** — первый нормированный раздел СП (structure-aware splitter уже бьёт по пунктам); таблицы-приложения — отдельный чанкинг строк. Карточка узла = шифр + название + область + список заголовков. Никакого LLM на построении каталога.

### Мультикласс/мультиузел — через диалог (ответ + варианты), не авто-fan-out

Когда стадия-0 видит несколько классов/узлов-кандидатов — система **не фанится во все сразу** (каша, цена, «полотно») и **не блокирует** жёстким `NEEDS_CLARIFICATION`. Вместо этого:
1. **Короткий ориентировочный ответ от верхнего класса/узла** (по уверенности), в форме под интент (слой 2).
2. **Остальные кандидаты — как предложенные варианты-развилки** (`suggestions`): другие классы («✉ переписка», «📁 проект Банкрот · ЭОМ»), другие узлы внутри класса, смежные аспекты («➕ климат/электрика — другой класс»).
3. **Клик → перезапрос со скоупом на выбранный узел/класс** (разворот по требованию).

Переиспользуем готовое: чипы `_render_suggestions(meta)` в UI + `clarification_service`. Кандидаты со стадии-0 → `suggestions` в ответе.

Явный мультикласс (запрос «и в нормах, и в проекте») — фан-аут с ограниченным бюджетом: общий бюджет ~B фрагментов **делится** по выбранным (класс, узел) пропорционально уверенности (пол 2 на узел) — шире фан-аут ≠ длиннее ответ. Слияние разных классов — **через cross-encoder реранк (ADR-3)**: он оценивает релевантность пары (запрос, фрагмент) напрямую, скор сопоставим между классами (косинусы из разных коллекций — нет). Сборка ответа — **секциями по классам** («По нормам …», «В проекте …», «В переписке …»). Слабый класс молча выпадает.

**Бонус-цель (не сразу):** когда в ответе рядом норма и проект — показать **дельту** («СП 485 требует газовое ПТ — в проекте ЭОМ его нет»); мост к нормоконтролю/диффу (W11/W13).

### Слой 2 — формирование ответа по интенту

Форму диктует **тип вопроса**, детерминированно, до генерации. ЛЛМ наполняет заданный каркас, форму выбирает классификатор:

| Интент | Форма |
|---|---|
| «расскажи / основные требования» | сжатый перечень: `пункт нормы — одна строка сути`, без вступлений/закрывашек |
| «собери всё / подробно» | развёрнуто, по разделам |
| «да/нет / какое значение / ширина» | одна строка + ссылка на пункт |
| «состав/перечень разделов» | голый список (уже есть outline-роут) |

## Развилки и решения по умолчанию (оператор правит)

1. **Класс vs датасет:** класс — новая верхняя ось; датасеты схлопываются в **фасеты внутри классов** (NTD_FIRE → фасет «пожарная» класса «норматив»). _По умолчанию: да._
2. **Оси проекта:** объект и раздел — **две независимые оси-фасета** (можно «весь раздел ОВ по всем объектам»), не жёсткое дерево объект→раздел. _По умолчанию: фасеты._
3. **Мультикласс:** запрос может идти по нескольким классам, результаты сливаются с пометкой класса источника. _По умолчанию: да._
4. **Справочник разделов РД:** seed фиксированный по **ГОСТ Р 21.101** (перечень разделов ПД/РД), дополняется из реальных проектов. _По умолчанию: гибрид._

## Врастание без слома ядра (ADR-1)

Аддитивно, как outline-роут: стадии 0/1 встают **перед** существующим плоским ретривом в `retrieve_chat_chunks`. Нашёлся уверенный набор узлов → сужаем scope leaf-поиска (payload-фильтр по узлам). Низкая уверенность/пусто → **fallback на текущий плоский поиск** (ничего не ломается). Гибрид+реранк остаются leaf-поиском; меняется только их **охват** — с датасета на узлы.

## Критерий приёмки

Кейс «требования к серверным»: стадия-1 поднимает СП 485/486/484 как узлы; ответ цитирует их пункты/приложения; форма — сжатый чек-лист. Метрика: «попал ли в нужные документы» на наборе доменных кейсов (серверные, эвакуация, огнестойкость).

## Заголовок документа — регион, не позиция (урок RAGFlow/DeepDoc)

Наблюдение оператора: **заголовок есть у любого документа, но не всегда в начале** — у норм после титула/регистрации, **у чертежей — в штампе (основной надписи) снизу справа** по ГОСТ 21.101. Карточка узла (стадия-1) не должна строиться из «первого чанка»: PoC 2026-06-17 взял `chunk_ord=0` у СП 485 и попал в глоссарий, а не в «Область применения» → СП 485 не поднялся в роутинге.

Референс — **RAGFlow DeepDoc**: layout-модель классифицирует регион `Title` по **визуальному типу с bbox-координатами**, а не по «верху страницы», поэтому штамп чертежа снизу опознаётся как заголовок. AnythingLLM — анти-пример (title = имя файла, плоский текст).

Следствия для каталога (стадия-1):
- **норматив:** карточка = название + раздел **«Область применения/Назначение»** (искать по `section_heading`, не `chunk_ord=0`); + шифр (regex).
- **чертёж/проект:** карточка = поля **штампа** (наименование, шифр листа, раздел) — извлекать как документ-уровневый чанк из основной надписи, не из тела. Проверить, тянет ли это CAD/BIM-экстрактор (греп 2026-06-17 по «штамп/title_block» — пусто; вероятно пробел).
- **чанк несёт позиционные метаданные** (страница + регион-тип) — для привязки ответа к месту и отделения штампа/таблиц от тела.

Прочее из RAGFlow, согласное с ADR (брать постепенно): типизированный чанкинг под класс (а не один General), **TSR — табличная структура** для смет/спецификаций/ВОР и **табличных приложений СП** (ровно то, что плоский поиск не достаёт в кейсе серверных), гибрид BM25+vector (уже есть, ADR-4) + cross-encoder rerank (есть, ADR-3) + document-level metadata-фильтры с операторами, «flattened-RAPTOR» (summary-чанк на уровень) вместо жёсткого drill-down.

## Вне области ADR

Перестройка индексации (чанкинг таблиц-приложений — связано с [ADR-5](LES3_PLAN.md) layout-aware парсер); класс «договор» как датасет; UI выбора класса/узла в Совушке (GUI-first — отдельно).
