# Аудит архитектуры РАГ: корректна ли, и почему «срезаемся на элементарном»

**Вопрос Олега:** правильная ли сама архитектура РАГ — если на «требования к шумоглушению» (а СП
есть в корпусе) мы спотыкаемся?

**Короткий вердикт:** ЯДРО корректное и современное (dense + learned-sparse + cross-encoder rerank).
Но оно обёрнуто в **~5 независимых quality/verification-гейтов + 3 роутера**, и они НЕ аддитивны —
они БЛОКИРУЮТ элементарный путь и компаундятся. Архитектура не «неправильная», она **переусложнена
и мискалибрована** — та же болезнь, что в [[rag-over-presets-not-keyword-channels]]/AUDIT_DETERMINISM:
то, что должно ПОМОГАТЬ, стало ГЕЙТАМИ, которые валят базовый случай.

## Полный путь «запрос → ответ» (как сейчас)

```
вопрос
 1. det-channels каскад (11 keyword) + agent_router         ← роутинг (см. AUDIT_DETERMINISM)
 2. SCOPE: route_query keyword → dataset_filter (NTD_HVAC) → resolve по ИМЕНИ датасета
        промах имени → было [] (пусто = «нет данных»)  ← ПОФИКШЕНО (→ весь NTD-корпус)
 3. RETRIEVE (retrieve_chat_chunks):
        • evaluate_retrieval_quality  ← ГЕЙТ: может занулить чанки
        • vector (Qdrant dense) + sparse (BGE-M3) → RRF → cross-encoder rerank   ← ЯДРО, ОК
        • doc_filter (route_documents — ЕЩЁ один LLM-роутер) + table_appendix (Ц9)
 4. GENERATE — 2 попытки:
        попытка 1: обычный sys (ирония) + llm_chunks
        пусто → попытка 2: sys_strict («ТОЛЬКО дословно, без обобщений»)
                 + concentrate_sources(min_score=0.5)  ← ГЕЙТ: ОТСЕКАЕТ чанки <0.5 → строгий промпт
                 без контекста → шаблонный отказ «нет точных данных»
 5. CRAG quality_status (VERIFIED/UNVALIDATED/HALLUCINATION/NO_DATA)   ← ГЕЙТ
 6. Т.О.С.К.А. validator (пост-ответ: answer vs context, может понизить)  ← ГЕЙТ
 7. semantic_cache · answer_form (keyword → max_tokens/style)
```

## Где «срезаемся на элементарном» (трасса «шумоглушение»)

1. scope keyword `NTD_HVAC` → нет такого датасета (есть NTD_GENERAL/CONSTRUCTION/FIRE/…) → пусто. **(пофикшено)**
2. retrieval теперь даёт **3 верных СП** (271 Шумоглушение, 60, 353).
3. НО попытка 1 пустая → попытка 2: `concentrate_sources(min_score=0.5)` выбрасывает релевантные
   чанки с умеренным скором + `sys_strict` «без обобщений» → **отказ при хороших источниках**.

Элементарный кейс умирает не в ОДНОМ месте, а от КОМПАУНДА гейтов: каждый по отдельности «разумен»,
вместе — глушат базовый ответ.

## Диагноз гейтов: что блокирует базовый путь

| Слой | Назначение | Проблема |
|---|---|---|
| scope-by-keyword-name | сузить на нужный корпус | хрупко: целит в несуществующее имя → пусто (пофикшено → весь NTD) |
| evaluate_retrieval_quality | отсечь мусорный ретрив | может занулить релевантное на умеренных скорах |
| 2-попытки + concentrate(0.5) + sys_strict | анти-галлюцинация | ОТСЕКАЕТ релевантные чанки и ЗАПРЕЩАЕТ синтез → отказ вместо ответа |
| CRAG + Т.О.С.К.А. validator | проверка ответа | ещё две точки, где ответ душится/понижается |
| dense+sparse+rerank | НАЙТИ | **корректно, это ядро, не трогать** |

## Что КОРРЕКТНО (не ломать)
Гибридный ретрив **dense + learned-sparse (BGE-M3) → RRF → cross-encoder rerank** — современно и
правильно. Чанкинг в токенах (ADR-7), типизированные приложения (Ç9) — тоже. **Проблема не в том,
КАК ищем, а в том, что НАД поиском навешано слишком много блокирующих гейтов.**

## Принцип исправления: элементарный путь РОБАСТЕН по умолчанию

> Нашли релевантные чанки → синтезировали ответ из них с цитатами. Это ДОЛЖНО работать всегда.
> Всё остальное (quality-гейты, CRAG, валидатор, strict-retry, doc-router, кэш) — **АДДИТИВНО**:
> улучшает, но НИКОГДА не зануляет/не отказывает, когда релевантные чанки есть.

Конкретно:
1. **scope**: промах → весь нормативный корпус, не пустота. **(сделано)**
2. **2-попытки**: `concentrate_sources` НЕ должен выбрасывать релевантные чанки (убрать жёсткий
   `min_score=0.5`-отсек или опустить); `sys_strict` смягчить — «отвечай НА ОСНОВЕ контекста, можно
   обобщать и формулировать, но не выдумывай факты вне него; цитируй пункт/СП». Синтез вместо отказа.
3. **evaluate_retrieval_quality / CRAG / валидатор**: не ОБНУЛЯТЬ ответ, когда rerank дал релевантное;
   максимум — помечать «не выверено», но ОТДАВАТЬ ответ с источниками.
4. Анти-галлюцинация остаётся **через цитаты + валидатор-как-метку**, а не через отказ-по-умолчанию.

## Гейт безопасности
Каждое смягчение — с golden 16/16 (retrieval) И ручной проверкой, что НЕ появились выдумки: ответ
обязан опираться на показанные источники. Балланс «не выдумывай ↔ синтезируй» держим цитатами.
