# План доделки ЛЕСа — цели (2026-06-22)

Формат: **1 цель — 1 проход**. Каждая цель самодостаточна и закрывается за один заход
(цель · шаги · готово-когда · зависит-от). Принцип прежний — ADR-11 (LLM последний, числа считает код).

---

### Ц1. Характер Совушки в детерминированных каналах
- **Цель:** вернуть тон. Каналы glossary/smeta/help/сверка отвечают плоско («эксель»), потому что
  идут мимо LLM, которая несла иронию. Нужен лёгкий **детерминированный flavor-слой**: короткая
  ироничная рамка вокруг фактов по персоне сыча («изящное хамство», числа строгие) — [[sovushka-tone-and-memory]].
- **Шаги:** `proxy/services/sovushka_tone.py` (пулы коротких реплик по типу ответа, выбор стабильный
  по хэшу вопроса — без рандома, тестируемо) → обернуть ответы каналов; числа/факты не трогать.
- **Готово, когда:** ответы звучат как Совушка, а не выгрузка; тесты на точность фактов зелёные.
- **Зависит от:** —  · быстрый проход.

### Ц2. Deploy-скрипт dev → рантайм
- **Цель:** убрать ручные `cp`+Edit по дивергентному клону (я так делал всю сессию — хрупко).
- **Шаги:** `tools/deploy_to_runtime.py` — порт изменённых файлов из LES_v2 в `/Users/ovc/LES`
  (новые — копия; модифицированные дивергентные — дифф/патч с предупреждением) + точечный
  `launchctl kickstart` нужных сервисов; dry-run по умолчанию.
- **Готово, когда:** «выкатить» = одна команда, без ручных патчей app.py/chat.py.
- **Зависит от:** —  · фундамент для всех следующих выкаток.

### Ц3. Плист-дрейф (КРИТ — корень «танцев»)
- **Цель:** `start_service` ставит репо-плист (cwd=проект) поверх рантайм-плиста → MLX/эмбеддер мрут.
- **Шаги:** параметризовать шаблоны плистов `LES_HOME` вместо хардкода; не перезаписывать рабочий
  плист (ставить только если отсутствует / по флагу). См. AUDIT_CORE §0,§3.1.
- **Готово, когда:** рестарт из любого клона не ломает MLX.
- **Зависит от:** —

### Ц4. Registry ↔ Qdrant: вернуть dataset_id
- **Цель:** каталог «Каталоги» = 0 хитов — реестровая запись `dataset_id=None`, фильтр не находит точки.
- **Шаги:** починить привязку id в реестре (или ре-тег точек). AUDIT_CORE §2.
- **Готово, когда:** «Каталоги» отвечает на запросы.
- **Зависит от:** —

### Ц5. База ГЭСН-2022: импорт выгрузки
- **Цель:** сборка ЛСР по ЛЮБОМУ коду, не только эталонному семени.
- **Шаги:** дать XLSX-выгрузку ресурсной части (ГРАНД-Смета/НСИ) → `tools/gesn_import` (готов) →
  parquet; при нужде подправить алиасы столбцов под конкретный экспорт.
- **Готово, когда:** «собери <любой код> объём N» считает из базы.
- **Зависит от:** **файл от Олега** (бесплатной машинной выгрузки нет).

### Ц6. `table_query` → SQL (DuckDB)
- **Цель:** заменить substring-скан Parquet на настоящий SQL — агрегации/джойны (сумма по разделу,
  смета×цены×объёмы). Фундамент под автосборку смет и аналитику РАГ.
- **Шаги:** DuckDB поверх Parquet (embedded, один дип); переписать `table_query_service` на SQL,
  fallback на старый путь.
- **Готово, когда:** «сумма по разделу X» считается SQL-запросом.
- **Зависит от:** —  (решение про duckdb уже принято — пора).

### Ц7. КАЦ из PDF-КП
- **Цель:** котировки не вручную. Vision-извлечение позиций+цен из КП (`Прайсы.rar`, 3 поставщика).
- **Шаги:** пайплайн стройсканов (OCR/vision) → строки котировок → `kac_service.analyze_kac`.
- **Готово, когда:** папка КП → КАЦ-таблица с выбором без ручного ввода.
- **Зависит от:** —

### Ц8. Рендер сметы в скачиваемый бланк
- **Цель:** собранный ЛСР → xlsx-выгрузка (наш бланк; GRAND-форма Прил.3/4 на 179 кол. — отдельно).
- **Шаги:** `to_xlsx` для результата `assemble`; кнопка/эндпоинт скачивания.
- **Готово, когда:** `/api/lsr/assemble` отдаёт скачиваемый файл.
- **Зависит от:** —

### Ц9. ADR-12: типизированный ретрив (табличные приложения)
- **Цель:** табличные приложения СП не поднимаются (кейс «серверные»: СП 485/486).
- **Шаги:** достроить спуск класс→узел→лист + каталог узлов для табличных приложений. ADR-12.
- **Готово, когда:** «серверные» поднимает нужный пункт СП.
- **Зависит от:** —  · крупный проход.

### Ц10. ADR-7: чанки в токенах
- **Цель:** qwen-профиль 1550 симв > 512 токенов → хвосты молча теряются.
- **Шаги:** резать по токенам, жёсткая проверка ≤ seq_len эмбеддера при старте.
- **Готово, когда:** нет потери хвостов на длинных чанках.
- **Зависит от:** —

### Ц11. ADR-5: layout-aware PDF
- **Цель:** кросс-страничные таблицы спецификаций (Docling/MinerU), pymupdf — fallback.
- **Готово, когда:** таблица-через-страницу извлекается целиком.
- **Зависит от:** —  · низкий приоритет.

### Ц12. Harvest → бенч → решение по LoRA
- **Цель:** копить правки оператора (verify) → `extract_bench` → go/no-go по VL-LoRA (не раньше).
- **Готово, когда:** есть таксономия ошибок и обоснованное решение.
- **Зависит от:** накопленная разметка (петля готова).

### Ц13. Бенч VL-модели

## Рекомендуемый порядок

Быстрый старт и фундамент: **Ц1 (характер)** → **Ц2 (deploy-скрипт)** → **Ц3/Ц4 (плист/registry)**.
Затем разблокировка: **Ц6 (DuckDB)** — открывает автосборку смет и аналитику. Параллельно ждём
файл ГЭСН (**Ц5**). Дальше — сметные хвосты (Ц7, Ц8) и качество РАГ (Ц9, Ц10).

Зависимости: автосборка по всей базе ← Ц5 (файл) + Ц6 (SQL). Безопасная выкатка всего ← Ц2.
