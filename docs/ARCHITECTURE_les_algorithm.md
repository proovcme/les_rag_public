# ЛЕС — общий алгоритм и архитектура (на критику)

> Документ для внешнего ревью. Цель — описать, **как ЛЕС обрабатывает запрос**, какой
> целевой костяк и где он расходится с текущей реализацией. Писалось честно: где демо —
> там «демо», где хардкод — там «хардкод». Раздел «Слабые места» в конце — специально под
> разнос. Стек: локальный RAG над строительными нормами/сметами, один пользователь, Mac M4
> (24 ГБ), движок MLX + Qdrant + SQLite. Прода нет, стадия разработки.

---

## 1. Принципы (откуда растут решения)

1. **Лестница инструментов (минимализм).** ИИ — последний инструмент, не первый. Сложность
   нарастает: `алгоритм → словарь → малая модель → локальная LLM → облако`. На каждую
   ступень поднимаемся, только если предыдущая не дотянула.
2. **Детерминизм — в инструментах, не в модели.** Число никогда не «генерится» языковой
   моделью. Оно либо **извлекается** (RAG из реальной нормы), либо **вычисляется** (код по
   формуле). Модель оркеструет и ставит задачи инструментам, но сама числа не выдаёт.
   Правило «числа строгие»: ирония/проза — можно вольно, цифры/нормы/пункты — кремень.
3. **Намерение объявляется, а не угадывается — когда возможно.** Если пользователь сказал,
   что хочет (режим/команда), не тратим вызов модели на угадывание. Угадывание — фолбэк
   для свободного ввода.
4. **Заземление обязательно для фактов.** Ответ по предметке = только из извлечённого
   контекста, с цитатами. Нет в контексте — «нет данных», без додумывания.

---

## 2. Целевой костяк: диспетчер → профиль → инструменты

Одна входная модель-**диспетчер** выбирает не ответ, а **профиль исполнения**.

```
                  ┌──────────────────────────────┐
   запрос  ──────▶│  ROUTER (малая, быстрая)     │  не отвечает —
                  │  классифицирует → профиль     │  выбирает конфигурацию
                  └───────────────┬──────────────┘
                                  │ ПРОФИЛЬ = {модель · промпт/роль · инструменты}
         ┌────────────────────────┼─────────────────────────┐
         ▼                        ▼                          ▼
  [САМ · малая ·           [САМ · малая ·            [ЭСКАЛАЦИЯ · большая/облако ·
   роль А · tools{RAG}]     роль Б · tools{норма,    роль В · tools{RAG,vision,
                            счёт,журнал}]            декомпозиция}]
         └────────────────────────┴─────────────────────────┘
                                  │  ДЕТЕРМИНИЗМ живёт здесь:
                                  ▼  RAG заземляет факт, код считает число
                            ответ + цитаты + вердикт валидации
```

**Профиль = тройка** `{модель-исполнитель · системный промпт+роль · набор инструментов}`.
Это то, что в текущем коде называется «режим» — но в целевой схеме это **выход router'а**,
а не кнопка. Развилка router'а добавляет ось «модель»: **сам** (дёшево, малая модель с
нужным промптом+инструментами) против **эскалация** (дорого, большая модель — только для
тяжёлого рассуждения/декомпозиции). Человек может **форсить** профиль (явный режим = «не
угадывай»); дефолт — router решает.

Детерминизм — свойство **инструментов** профиля, а не модели. Любая модель в любом профиле
только оркеструет: понимает вход, раскладывает задачу, **ставит подзадачи инструментам**,
собирает результат. Финальное число приходит из инструмента (нормы/формулы), не из токенов.

## 3. Полный алгоритм обработки запроса (текущая реализация)

`_run_chat(request)` в `proxy/routers/chat.py`. Запрос: `{question, mode?, project_id?,
dataset_ids?, dataset_filter?, output_directive?, validation_enabled?}`. Стрим и нестрим
делят одно ядро (`token_sink`).

### 3.1. Префильтры (детерминированные, до моделей)
1. **Команды** (`/`-палитра, `command_service`): `rewrite` → переформулировать и пойти
   дальше; иначе — детерминированный ответ сразу.
2. **Явный режим** (`mode`, форс минуя router/RAG):
   - `smeta` → `object_estimate` напрямую (regex-парс → шаблон → формулы → ГЭСН → ЛСР). **0 LLM.**
   - `review` → `run_normcontrol` (форматы/шифры/комплектность PDF). **0 LLM.**
   - `kp` → заглушка (генерация КП — задел). **0 LLM.**
   - `rag` → форс заземлённого RAG (гейтит router+каскад).
   - `free` → прямой LLM **без** ретрива + плашка «вольно».

### 3.2. Маршрутизация (если режим не задан)
3. **Router-модель** (`agent_router_service`, флаг `LES_ROUTER_PRIMARY`, сейчас на облаке
   ~1с): классифицирует запрос в инструмент из реестра (asbuilt/les_md/registry/glossary/
   смета-семья/table_agg/clause/memory/decision/none…). Вернул не-none → детерминированный
   ответ через соответствующий хэндлер. **Сейчас router выбирает только инструмент, не
   профиль целиком — это главный разрыв с целевой схемой (см. §7).**
4. **Каскад keyword-каналов** (фолбэк, если router=none/сбой): regex+SQL, первый сработавший.
5. **Автозаметка** (утверждение-факт → запоминаем, 0 LLM).
6. Ничего детерминированного не сработало → **RAG-конвейер**.

### 3.3. RAG-конвейер (заземлённый ответ)
7. **Скоуп датасетов**: явный выбор > режим проекта (project_id → датасеты объекта) > весь корпус.
8. **Reconcile / clarification**: сверка ВОР↔КС-2↔смета (детерм.); решение «не слишком ли
   широк запрос» (saferag) → может попросить уточнить.
9. **Ретрив** (`retrieve_chat_chunks`), под-фазы:
   - **doc_router** (типизированный ретрив, ADR-12): LLM-роутер по каталогу документов →
     сузить поиск до узлов-документов (кэшируется).
   - **vector**: эмбеддинг запроса (Qwen3-Embedding-0.6B, CoreML, 1024-dim) → Qdrant ANN
     (Cosine, payload-фильтр по dataset_id/file_name).
   - **sparse**: BGE-M3 learned-sparse (Qdrant-native, за флагом).
   - **lexical**: FTS5 по SQLite (точные термины/коды норм).
   - **RRF-merge** трёх сигналов → **дедуп по content_hash** → **cross-encoder реранк**
     (bge-reranker-v2-m3) → опц. подъём табличных приложений норм.
10. **Контекст-окна** (`expand_context_windows`): для каждого чанка дочитать соседей/родителя
    из lexical SQLite (одно соединение, индекс по collection/dataset/doc/ord).
11. **Сборка контекста** (`build_context`): склейка с метаданными, чистка `<br>`-тег-супа,
    лимит символов.
12. **Генерация** (LLM, стрим): `sys_normal` (заземление + персона + «оформляй таблицей»)
    + каркас формы под интент (`answer_form`) + формат-директива из GUI. Облако — цепочка
    моделей с таймаутом и фолбэком; локаль — MLX (Qwen3.5-4B). До 2 попыток (ретрай —
    строже промпт, шире пул).
13. **Валидация** (TOSKA, `/api/validate`, coreml/MLX): аддитивна — метит
    VERIFIED/NO_DATA/HALLUCINATION, **не форсит** ретрай (fail-open). Вердикт → `crag_status`.
14. **Ответ**: `{answer, sources, crag_status, query_route, retrieval_trace, validation,
    history_id}`. История пишется всегда (видна в GUI).

## 4. Слой инструментов (детерминированные движки)

- **Сметный движок** (`object_estimate_service`, `lsr_assembly_service`, `gesn_service`,
  `nr_sp_service`, `fgis_price_fetch_service`): фраза → ВОР → позиции → ОЗП/НР/СП/цена →
  итог. **Источник декомпозиции сейчас — захардкоженный YAML-шаблон** (2 штуки: дерев. дом,
  монолит-офис). Это демо; на незнакомый объект (паркинг, мост) честно «нет шаблона».
- **Нормоконтроль** (`normcontrol_service`): формальные проверки PDF (форматы, шифры,
  ведомость↔файлы). 0 LLM.
- **Табличная агрегация** (`table_query_service`, parquet): суммы/итоги по ведомостям.
- **Извлечение по схеме** (`extract_service`): схема → валидный JSON (облако native, локаль
  validate-and-repair).
- **Память/задачи/решения/журнал** (regex+SQLite): команды без модели.

## 5. Слой данных

- **Qdrant** `les_rag_qwen3_06b`: 179 424 точки, 1024-dim Cosine, payload-индексы
  `dataset_id`/`file_name`, parent/child-чанкинг (медиана 896 симв, 100% метаданных).
- **Lexical SQLite**: FTS5 (unicode61) + таблица чанков с индексами под context-окна.
- **ГЭСН-2022 parquet**: 42 408 норм / 609 987 строк-ресурсов (офиц. ФГИС ЦС).
- **YAML-шаблоны объектов**, **онтология терминов**, **session/work memory** (SQLite).

Перф после индекс-фиксов: context-фаза 3.75с→**0.008с**, ретрив 1.9с→**0.4с**. Узкое
место — генерация LLM (~5с, стримится).

## 6. Где модель, где код (разделение ответственности)

| Задача | Кто | Почему |
|---|---|---|
| Понять messy-запрос, классифицировать | модель (router) | язык, неоднозначность |
| Разложить объект в ВОР, поставить задачи | **модель (цель; сейчас — хардкод YAML)** | понимание объекта |
| Выбрать норму под позицию | RAG + код (`get_norm`) | заземление |
| Посчитать объём/цену/итог | код (формулы, ЛСР) | детерминизм |
| Синтез ответа из источников | модель | язык |
| Проверить ответ на галлюцинацию | валидатор (отд. модель) | независимость |
| Финальное число | **никогда не модель** | «числа строгие» |

## 7. Текущее ≠ целевое (честный разрыв)

1. **Router выбирает инструмент, а не профиль.** Нет оси «модель-исполнитель: сам/эскалация»
   и привязки {промпт+tools} к маршруту. Режимы сейчас — **хардкод-кнопки в GUI** + ранний
   `if mode==…` в `_run_chat`, а не выходы router'а.
2. **Декомпозиция объекта захардкожена** (YAML-шаблоны). Целевое — LLM+RAG раскладывают
   объект в ВОР и ставят задачи существующему движку. Шаблонный подход растёт линейно и
   длинный хвост не покрывает.
3. **Эскалации как явного решения нет.** «Облако vs локаль» сейчас — глобальный флаг
   провайдера, а не пер-запросное решение router'а по сложности.
4. **Профили не выделены в сущность.** {модель × роль × инструменты} размазано по коду
   (sys_normal, answer_form, режимы, каскад), а не библиотека профилей.

## 8. Слабые места и открытые вопросы (под разнос)

1. **Двойная маршрутизация.** router-модель (облако) + keyword-каскад + явные режимы —
   три механизма выбора пути, частично перекрываются. Не слишком ли? Не схлопнуть ли в один
   router-профиль?
2. **Дедуп инертен.** Дедуп по content_hash на боевом пути ни разу не сработал (скоуп сужает
   раньше). Нужен или нет?
3. **Качество контента**: ~18% битых таблиц (`<br>`-суп), ~16% англ. Revit-шум, кросс-
   датасетные дубли с одним хэшем, семантические ямы на узких интентах. Чистится курированием
   — но это ручной долг, не алгоритм.
4. **Валидатор fail-open** (точность ~25%, метит но не блокирует). Это безопасно (не режет
   верное), но и пропускает галлюцинации. Где баланс?
5. **doc_router (типизированный ретрив)** добавляет LLM-вызов в каждый предметный запрос.
   Окупается ли качеством против латентности/стоимости?
6. **Эскалация в облако** упирается в сеть рантайма (Cloudflare/Google режутся; OpenAI
   direct работает). Архитектура завязана на доступность облака для «тяжёлых» профилей.
7. **Один пользователь, одна машина (24 ГБ).** Любая «вторая модель» (отдельный эмбеддер,
   валидатор, реранкер) конкурирует за RAM/ANE. Сколько моделей реально держать тёплыми?
8. **Шаблонная смета выдаёт строгое число поверх геройских допущений** (квадратное пятно,
   5 позиций, без инженерки/отделки/земли). Строгая арифметика над кривой моделью объекта —
   это строгая ошибка. Целевая декомпозиция через LLM это снимает, но не реализована.
9. **Нет петли «модель ставит задачу инструменту → инструмент отвечает → модель уточняет».**
   Сейчас потоки в основном одно-проходные (router → хэндлер → ответ). Tool-use/agentic-цикл
   как первоклассный механизм отсутствует.

## 9. Резюме для ревьюера

ЛЕС сегодня — **заземлённый RAG с детерминированными движками и тройной маршрутизацией**
(router-модель + keyword-каскад + явные режимы). Костяк здоров: нарезка/индекс/заземление/
валидация на месте, перф вылечен. Главный архитектурный долг — **схлопнуть три механизма
выбора пути в один router, выдающий полный профиль `{модель × роль × инструменты}`, с явной
эскалацией по сложности**, и **заменить захардкоженную декомпозицию (YAML-шаблоны) на
LLM+RAG, ставящие задачи существующему детерминированному движку**. Тогда «режимы» исчезают
как класс, а длинный хвост объектов покрывается пониманием поверх живой базы норм, не руками.

Грызите по §7 и §8.

## 10. Поправки по итогам внешнего ревью (Codex)

Ревью било точно. Ниже — что принято в архитектуру (с уточнением формулировок), и что
сознательно отложено как золочение для dev-стадии (один пользователь, 24 ГБ, прода нет).

### 10.1. Три коренные поправки (приняты целиком)

**(A) «Один router» = единый КОНТРАКТ маршрутизации, а не одна LLM.** Явный режим, команда,
regex, keyword-каскад и LLM-router — это разные **источники** одного решения, идущие по
приоритету (cascade-of-confidence). Они не уничтожаются; они приводятся к одному контракту
`ProfileResolution`. Пользовательская кнопка-режим — не мусор, а явный override/hint.

```
explicit mode/command → deterministic recognizers → cheap keyword → LLM router → fallback
```
```jsonc
ProfileResolution = {
  "profile_id": "norm_grounded_answer",
  "route_source": "explicit_mode | command | regex | keyword | llm_router | fallback",
  "confidence": 0.86,
  "workflow_id": "…",            // router выбирает ЗАДАЧУ/workflow, а не атомарный инструмент
  "required_slots": ["dataset_scope"],
  "missing_slots": [],
  "escalation_allowed": false,
  "reasons": ["…"]
}
```
Инвариант: **router сам пользователю не отвечает** — только выбирает профиль.

**(B) Число «строгое» только относительно ЯВНО указанной базы.** Строгая арифметика поверх
выдуманного состава работ = строго оформленная ошибка. Каждое число несёт provenance:

```
EXTRACTED  — из источника (RAG)          COMPUTED  — по формуле из входов
ASSUMED    — принято как допущение        ESTIMATED — по шаблону/аналогии
USER_GIVEN — дал пользователь             UNSUPPORTED — нет основания
```
Число в ответе — не `value`, а `{value, unit, kind, formula?, inputs[], basis[],
assumptions[], confidence}`. Тогда ответ звучит не «1 284 500 ₽», а «1 284 500 ₽ при
допущениях: квадратное пятно, без инженерки/отделки, шаблон v1». (Старт — provenance на
уровне сметы целиком: basis+assumptions; пер-числовой граф inputs — v2, не для 2 шаблонов.)

**(C) Не «LLM-декомпозиция», а SCHEMA-FIRST декомпозиция с валидацией.** LLM не строит смету
напрямую (это вернёт фантазию). LLM предлагает **кандидата объектной модели в строгой схеме**;
код валидирует; недостающее — спрашивается или метится допущением.

```
описание → LLM извлекает параметры объекта в СХЕМУ (object_type, этажи±, площадь,
           конструктив, включённые/исключённые разделы, missing_inputs[])
         → система ловит missing slots → RAG достаёт нормы/аналоги/семьи работ
         → LLM предлагает кандидатов ВОР (в схеме) → ДЕТЕРМ. ВАЛИДАТОР:
           единицы, обяз. размеры, дубли работ, несовместимые работы, нет семей,
           есть ли норма → clarification / пометка допущений → движок считает → ответ с допущениями
```
LLM предлагает, RAG заземляет, код валидирует, пользователь видит все допущения.

### 10.2. Принято как костяк рефакторинга (порядок — поведение НЕ ломаем)

1. **`Profile` — декларативная сущность** (не только `модель×роль×инструменты`, а ещё
   `workflow, retrieval_policy, validation_policy, escalation_policy, failure_policy,
   output_contract, latency_budget, models_required`).
2. **`ProfileResolver`** объединяет все источники маршрутизации в `ProfileResolution`.
3. **Текущие `mode==…` → профили** без смены поведения (порт, не переписывание).
4. **`ExecutionPlan`/workflow**: router выбирает задачу, workflow дёргает инструменты.
5. **Claim-level валидация** вместо общего hallucination-score: три проверки —
   `citation_coverage` (нормативное утверждение → источник), `numeric_provenance` (число →
   извлечение/формула/допущение), `context_entailment` (не утверждать больше контекста).
   TOSKA остаётся доп-сигналом, не главным механизмом доверия.
6. **Типизированная память**: `user_assertion / verified_project_fact / system_inference /
   assumption / decision / temporary_context`. **Непроверенная память НЕ участвует в расчёте
   молча** — подсказка, не основание.
7. **Bounded tool-loop** (НЕ свободный ReAct): `max_steps≤4`, белый список инструментов,
   stop-условия (`answer_ready/missing_input/no_data/conflict`), запрет
   `invent_norm_code/invent_quantity`. Типизированные контракты tool-call и tool-result.
   Делать ПОСЛЕ schema-first (иначе цикл нечего планировать).
8. **Audit-trace с версиями**: `prompt_version, profile_version, gesn_version,
   price_base_date, embedding_model, reranker_model, norm_base_version` — чтобы через месяц
   объяснить, почему тот же запрос дал другой ответ.

### 10.3. Инварианты системы (новый контракт)

1. Любое нормативное утверждение имеет источник.
2. Любое число — извлечено / вычислено / помечено допущением. Иначе `UNSUPPORTED`.
3. Неподтверждённое допущение не отображается как факт.
4. Router не отвечает пользователю — только выбирает профиль.
5. Инструмент возвращает структурный результат, не прозу.
6. Конфликт источников — показываем конфликт, не усредняем.
7. Не хватает входных данных — спрашиваем ИЛИ явно метим расчёт предварительным.
8. Валидация не ухудшает ответ молча: метит или блокирует по политике профиля.
9. Облачный профиль не получает проектный контекст без политики допуска.
10. Каждый расчёт имеет воспроизводимый calculation-trace.

### 10.4. Принято, но ОТЛОЖЕНО (right, но не для dev-стадии сейчас)

- **Cloud data-egress политика (L0–L4, редакция).** Верно и важно для прода; сейчас облако =
  глобальный флаг, проектный контекст в облако осознанно не уходит. Фиксируем как инвариант №9,
  реализация — к проду.
- **RAM-aware профили (resident set, eviction).** Реальное ограничение 24 ГБ, но это
  операционный тюнинг, не блокер архитектуры. Лёгкая версия: `max_loaded_models`, валидатор —
  первый на выгрузку (он fail-open и слабый).
- **doc_router — абляции (Recall@K/MRR/nDCG, latency/cost).** Принято как дисциплина, но нужен
  eval-набор; до него — `conditional use` (включать при широком scope / явном типе документа /
  отсутствии project_id / высокой confidence).
- **Ingestion QA report** (table_quality, language_ratio, ocr_noise, duplicate_cluster,
  metadata_completeness, revit_noise) — это и есть настоящий рычаг против контент-долга (18%
  битых таблиц, 16% шум, дубли). Отдельный workstream; **дедуп переезжает с hot-path в
  ingestion** (рантайм-дедуп оставить дешёвым guard'ом или снять).
- **Citation до строки таблицы** (table_id/row_id, не только chunk) — нужно для проверяемости
  норм-таблиц; зависит от ingestion-метаданных.

### 10.6. Что меняю первым (ревизия §19-порядка)

`Profile` + `ProfileResolution` (контракт) → порт текущих режимов в профили (0 поведения) →
numeric provenance на уровне сметы → claim-level валидация → типизированная память →
ingestion QA → **только потом** schema-first декомпозиция → bounded tool-loop. Eval-набор —
сквозной (нужен и для doc_router-абляций, и для валидации декомпозиции).

**Резюме после ревью (заменяет §9 в части «режимы исчезают»):** цель — не «схлопнуть всё в
router-модель» и не «заменить YAML на LLM-декомпозицию», а: **(1) свести все источники
маршрутизации к одному контракту `ProfileResolution`, где LLM-router — лишь один источник;
(2) заменить YAML на schema-first объектную модель (LLM предлагает, RAG заземляет, код
валидирует, допущения видны); (3) числа строгие ТОЛЬКО относительно явной базы — источник,
формула, входы, допущения, версия норм, статус проверки.** Режимы GUI остаются как явные
override/hint к профилям. Детерминированные движки — источник чисел. LLM — планировщик и
компоновщик, не источник фактов и расчётов.
