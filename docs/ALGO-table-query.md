# Алгоритм: счёт по ячейкам таблиц (table_query)

Как Л.Е.С. **правильно считает по таблицам** (суммы, количества) — детерминированно,
без участия LLM в арифметике. Принцип ADR-11: LLM — последний инструмент; числа считает
код ([[llm-minimalism-principle]]).

## Идея в одной фразе

**LLM решает, ЧТО считать; СКОЛЬКО — считает Python поверх Parquet.** Модель к числам
не прикасается (она в арифметике галлюцинирует). Ячейка таблицы превращается в типизированное
число, и агрегация идёт детерминированным кодом по полному набору строк.

Код: `proxy/services/table_query_service.py`, `backend/parquet_writer.py`,
роутинг — `proxy/services/query_router.py`.

## Почему обычный RAG считает НЕправильно

1. Таблица в эмбеддинге — это текст; модель «видит» куски и складывает на глаз → врёт.
2. RAG достаёт **top-k** релевантных чанков → даже если модель сложит идеально, у неё на
   руках лишь ЧАСТЬ строк → сумма частичная. Реальный кейс: кабель 3х1,5 выходил
   **5900 вместо 15030,72** — недостающие строки просто не попали в top-k.

## Шаги алгоритма

1. **Нормализация: ячейка → типизированное число.**
   `parquet_writer` читает таблицу (xlsx/pdf/docx), находит строку-заголовок
   (`_find_header_row`), маппит колонки в единую схему (`_map_columns_simple`:
   `qty`/`amount`/`amount_mat`/`amount_work`/`name`/`unit`/…) и сохраняет
   **одна строка таблицы = одна запись Parquet** с числовыми полями (`save_parquet`).
   *Грабли:* «Наименование … материалов» ловилось правилом `amount_mat` раньше `name`
   → имя терялось, строка выпадала. Фикс — приоритет «наименование»→`name`.

2. **Роутинг в табличный канал.** `query_router` ловит агрегатный вопрос (токены
   «сколько / итого / сумма / кол-во / объём / стоимость» + табличный контекст
   «смета / спецификация / кабель / …») и уводит в канал `table` (`maybe_answer_table_query`).
   Вызов гейтится router-интентом и происходит **после** ретрива; агрегация — пост-ретрив
   над Parquet задействованных датасетов, а не по retrieved-чанкам. Не похоже на табличный
   запрос → `None` → обычный RAG.

3. **Детерминированный выбор колонки (`_select_numeric_field`).**
   По словам вопроса: «метраж / сколько / объём / погон» → `qty`; «стоимость / руб /
   затраты / цена» → `amount`. Нет явного признака → `qty` (для ВОР/смет объём — основное).
   **Data-aware fallback:** если выбранная колонка пуста ИЛИ все нули (напр. ВОР с
   `amount=0`) → автоматически переключаемся на `qty`.

4. **Фильтр по предмету (`_query_keywords` + `_row_matches`).**
   Из вопроса достаём ключевые слова предмета («кабель 3х1,5»), суммируем **только
   совпавшие строки**, а не всю таблицу вслепую. `sum` без предмета не выполняется —
   отвечаем списком (чтобы не выдать бессмысленный итог).

5. **Полная агрегация по ВСЕМУ Parquet, а не по top-k.** Ключевой фикс: при `operation=sum`
   и наличии предмета берём **все строки Parquet** задействованных датасетов
   (`parquet_ref_chunks_for_datasets`, `max_rows=10**9`), а не retrieved-чанки. Затем
   `total = sum(numbers)` — чистый Python.

6. **Ответ + статус.** Число форматируется (`_sum_answer`) с источниками; `crag_status`
   = VERIFIED (значение проверяемо по Parquet, не сгенерировано моделью).

## Критерий приёмки (доказано вживую)

Кабель медный 3х1,5 ППГнг → **15 030,72 м**, бит-в-бит совпало с ручным расчётом из
исходного .xls (раньше RAG выдавал ~5900). Тесты: `tests/test_table_query_service.py`.

## Связанное

Тот же принцип («числа считает код по Parquet, не LLM») лежит в основе:
`bor_service` (свод ВОР), `spec_to_bor_service` ([[ALGO-spec-to-bor]]),
`reconcile_service` (сверка ВОР↔КС-2↔смета↔ИД), `plan_fact_service` (план/факт).

## Границы

- Считаем по тому, что распарсилось в Parquet. Кривой исходный заголовок/мердж ячеек →
  чинить в `parquet_writer` (header-детект / column-mapping), а не в агрегаторе.
- Нечисловые «итоги прописью», сложные многоуровневые шапки — вне области; нужен
  отдельный разбор шапки.
