# Пособие: как документировать код (для вайбкодеров)

> Портативное. Без привязки к проекту — копируй в любой репозиторий.
> Цель: чтобы документация **не врала** и через полгода ИИ-генерации, и чтобы новый человек или агент
> понял систему за 10 минут, а не за неделю археологии.

## Почему это вообще проблема (и почему ИИ её усиливает)

Вайбкодинг = быстро строить с ИИ. Побочка: ИИ генерит и код, и доки — но **доки дрейфуют быстрее кода**.
Через месяц у тебя:

- 70 markdown-файлов, половина — датированные «итоги сессии», которые никто не перечитывает;
- доки описывают функции, которых уже нет, и не знают про те, что есть;
- три разных «версии» проекта в трёх местах, и ты не понимаешь, где ты сейчас;
- каждый новый агент верит старому слою и ломает то, что чинили вчера.

**Корень:** документацию писали как НАРРАТИВ (что я сегодня сделал), а не как КАРТУ (что есть сейчас).
Карта стареет медленно; нарратив устаревает в момент написания. ИИ генерит нарратив охотно и много —
поэтому без дисциплины свалка растёт со скоростью генерации.

## Главный принцип: документация = код

Один закон, из которого следует всё остальное:

> **Док обязан соответствовать коду. При расхождении прав КОД. Любое утверждение в доке проверяемо
> против кода/тестов/рантайма.**

Если этого нет — документация хуже, чем её отсутствие: она активно врёт и тратит время.

## Семь правил

### 1. Один индекс, а не россыпь файлов
Заведи ОДИН файл-карту: каждый модуль — одна строка → назначение · точки входа (где код) · статус ·
ссылка на детальный док. Новый человек начинает отсюда. Это карта, не учебник.

### 2. Статус честный и видимый
У каждого дока/модуля — явная метка: **актуально / разошлось с кодом / историческое / план**. Метка
снимает главную боль: «можно ли этому верить?». Док без статуса по умолчанию подозрителен.

### 3. Один модуль — один док, по шаблону
Не один талмуд на всё. На каждый модуль — короткий док фиксированного формата:
`назначение · точки входа · данные/конфиг · зависимости · поток (с формулами/шагами) · статус-vs-код ·
грабли/границы · тесты`. Одинаковый шаблон = предсказуемость.

### 4. «Где мы сейчас» — отдельный леджер
Единый источник состояния: версия, что задеплоено, что вошло. Версии — **понятные и растущие**
(`0.МАЖОР.ФИЧА.ПАТЧ`), двигаются на каждую фичу. Если версия стоит на месте, пока летят коммиты —
ты ослеп. Не держи три несвязанные оси версий: одна основная, остальные — компоненты.

### 5. Архивируй, не удаляй
Историческое (датированные итоги, заменённые планы) → в `archive/` с индексом «что и почему здесь +
где текущая правда». Канон остаётся тонким. Ничего не теряется (и git хранит историю), но и не мусорит.

### 6. Не плоди датированные саммари
«Итог сессии от 12.03» — это нарратив, он мёртв при рождении. История изменений живёт в `git log` +
леджере. Хендофф — это не новый файл, это **обновлённые живые доки** (карта + леджер + бэклог).

### 7. Генерируемое — из источника
Если док генерится (из конфига, кода, схемы) — правь ИСТОЧНИК и регенерируй, не редактируй вывод руками.
Иначе при следующей генерации правка потеряется, и снова расхождение.

## Ритуал: что делать по ЛЮБОМУ изменению

Изменение «готово» не когда код работает, а когда:

```
[ ] код + тест на изменение (тест ловит регрессию)
[ ] обновлён док модуля (поток/границы/точки входа — если менялись)
[ ] обновлён индекс: статус модуля (актуально/разошлось) + новые точки входа
[ ] версия двинута + строка в леджер (что вошло)
[ ] гейт зелёный (синтаксис/импорт + тесты) ПЕРЕД «готово»/деплоем
[ ] в прод — только после гейта, с известным путём отката
```

Маленькое правило, экономящее недели: **док правится В ТОМ ЖЕ коммите, что и код.** Не «потом» — потом
не наступает.

## Анти-паттерны (красные флаги)

- **Док-нарратив вместо карты.** «Сегодня я добавил X» вместо «X есть, вот где, вот статус».
- **Метка врёт.** Коммит/док говорит «сделано/исправлено», а на деле нет. Метка хуже молчания.
- **Versioning-дрейф.** Версия в UI ≠ в пакете ≠ в доках; никто не знает текущую.
- **Призрачные ссылки.** Док ссылается на сервис/файл, которого нет (переименовали/удалили).
- **Дубли.** Два README, три overview, четыре плана — все слегка разные, все слегка устарели.
- **Talmud.** Один файл на 1500 строк «обо всём» — его не читают и не обновляют.
- **Хардкод в тестах того, что меняется** (версия-литерал) — тест краснеет на каждый бамп, его начинают игнорить.

## Проверка «репо сияет» (чек-лист здоровья)

```
[ ] есть ОДИН индекс модулей, и он совпадает с реальным кодом (счётчики, имена сервисов)
[ ] у каждого дока есть статус; «разошлось»/«историческое» помечены, не притворяются актуальными
[ ] видно «где мы»: одна понятная версия + что задеплоено
[ ] историческое — в archive/, канон тонкий
[ ] нет битых ссылок на перемещённые/удалённые файлы
[ ] есть док «что не ломать» (гейт в прод, риск-тиры, откат)
[ ] новый агент/человек, прочитав индекс, находит нужный модуль и его код за минуты
```

## Одна мысль на вынос

Документируй **состояние, а не события**. Карта + статус + леджер + ритуал «док в том же коммите».
Тогда документация переживает скорость ИИ-генерации и остаётся тем, чем должна быть — **картой правды**,
а не музеем благих намерений.
