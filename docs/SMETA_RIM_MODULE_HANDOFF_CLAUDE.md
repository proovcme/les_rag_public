### Что было сломано

Старая база ГЭСН смешивала нормы по голому номеру `38-01-001-01`.

Это критическая ошибка: `ГЭСН38-01-001-01` и `ГЭСНм38-01-001-01` — разные нормы.
До фикса `ГЭСНм38-01-001-01` мог получать ресурсы гидротехнической строительной нормы
и давать неверный расчёт.

### Как исправлено

Введён ключ нормы:

```text
norm_key = <base_type>:<bare_norm_code>
```

Примеры:

```text
ГЭСН:38-01-001-01
ГЭСНм:38-01-001-01
```

Затронутые файлы:

- `tools/gesn_import.py`
- `tools/gesn_pdf_import.py`
- `proxy/services/gesn_service.py`
- `tests/test_gesn_import.py`
- `tests/test_gesn_pdf_import.py`
- `tests/test_gesn_bulk_import.py`

Новая база:

- dev: `data/gesn_base/gesn2022_v2.parquet`
- runtime: `/Users/ovc/LES/data/gesn_base/gesn2022_v2.parquet`

Сервис читает старую базу как fallback и накрывает её v2-слоем. Не склеивать ресурсные строки
между старой и новой базой: новая норма должна заменять старую по `norm_key` целиком.

### FGIS-версия нормы

Для `ГЭСНм38-01-001-01` используется актуальная публикация ФГИС:

```text
documentTypeName = ГЭСНм
norm_key = ГЭСНм:38-01-001-01
id = 597835
source_guid = e97f70e5-6acc-4f6f-a587-149203cfd67e
```

FGIS SearchEstimatedRates может вернуть несколько публикаций одной таблицы. Парсер берёт
последнюю по максимальному `id` внутри одинаковых `(documentTypeName, documentName)`.
Не возвращать старый GUID `2233d1b3...` в расчёт столпа: он даёт старую ресурсную редакцию
и может породить дубли.

Актуальный состав нормы `ГЭСНм38-01-001-01`: 12 ресурсов, включая:

- труд `1-100-40`;
- агрегат `Затраты труда машинистов`;
- машины `91.05.05-015`, `91.14.02-002`, `91.17.04-042`, `91.17.04-233`, `91.21.12-002`, `91.21.16-012`;
- материалы `01.3.02.08-0001`, `01.3.02.09-0022`, `01.7.03.04-0001`, `01.7.11.07-0227`.

### ОТм / машинисты

ОТм больше не оставляется `missing`, если агрегатная строка `Затраты труда машинистов`
полностью покрывается известными машинами позиции.

В `proxy/services/lsr_assembly_service.py` добавлен минимальный доказанный маппинг:

```text
91.05.05-015 -> 4-100-060  # кран на автомобильном ходу 16 т, машинист 6,0
91.14.02-001 -> 4-100-040  # автомобиль бортовой, машинист 4,0
91.14.02-002 -> 4-100-040  # автомобиль бортовой до 8 т, машинист 4,0
```

Правило строгое:

- если сумма машино-часов известных машин совпадает с агрегатом ОТм;
- тогда агрегатная ОТм-строка заменяется тарифными строками машинистов;
- если покрытие неполное — оставить `missing`, не угадывать.

Для столпа:

```text
ОТм агрегат = 9.5 чел-ч/т
9.0 маш-ч/т кран 16 т -> 4-100-060
0.5 маш-ч/т бортовой авто до 8 т -> 4-100-040
```

В сплит-форме СПб 2 кв. 2026:

```text
4-100-060 = 801.41 руб/чел-ч
4-100-040 = 596.61 руб/чел-ч
```

Не расширять этот маппинг “по смыслу”. Только по ФСЭМ/официальному источнику или отдельному
проверенному fixture.

### Главная ловушка

Не делать “помощь LLM” через жёсткое угадывание:

- не выбирать норму по голому номеру;
- не подставлять среднюю ОТм;
- не склеивать старую и новую публикации ФГИС;
- не считать цены из текста;
- не скрывать `missing`.

Правильный принцип:

```text
модель связывает контекст -> код выбирает workflow -> справочник даёт норму/цену -> код считает -> missing остаётся видимым
```

## Зачем

В LES уже есть сметное ядро: ГЭСН разворачивается в ресурсы, ФГИС ЦС даёт цены, КАЦ закрывает отсутствующие материалы, `lsr_assembly_service` считает ОЗП / ЭМ / М / ФОТ / НР / СП / итог.

Проблема была не в отсутствии арифметики, а в отсутствии методической трассы РИМ по графам ЛСР: системе нужно не просто сказать сумму, а показать, откуда взялась каждая строка и каждая графа Приложения 3 к Методике 421/пр.

## Что добавлено

Новый сервис:

- `proxy/services/rim_lsr_trace_service.py`

Назначение:

- строит доказательную трассу одной позиции ЛСР;
- использует существующий `gesn_service` для нормы и ресурсов;
- использует существующий контракт `PriceBook.lookup()` ФГИС ЦС;
- сохраняет происхождение цены:
  - `fgis_current` — графа 10 из текущей цены сплит-формы;
  - `fgis_base_index` — графа 8 = базовая цена, графа 9 = индекс, графа 10 = графа 8 × графа 9;
  - `manual` — явная цена в ресурсе;
  - `kac` — цена КАЦ;
  - `missing` — нужен добор цены / КАЦ;
- применяет коэффициенты по виду ресурса:
  - labor → ОЗП;
  - machine / machinist → ЭМ и ОТм;
  - material → без коэффициента;
- формирует строки:
  - работа;
  - коэффициент;
  - ОТ(ЗТ);
  - детализация трудозатрат;
  - ЭМ;
  - машины;
  - ОТм(ЗТм);
  - машинисты;
  - М;
  - материалы;
  - прямые;
  - ФОТ;
  - НР;
  - СП;
  - итог позиции.

Тесты:

- `tests/test_rim_lsr_trace_service.py`

Покрыто:

- эталонная позиция `ГЭСН12-01-034-02` даёт итог `11813.04`;
- графы 2 / 4 / 7 / 12 проверяются на ключевых строках;
- коэффициенты применяются к ОЗП / ЭМ / ОТм, но не к материалам;
- различается цена из текущей графы сплит-формы и цена `база × индекс`.

Ключевые правила из `алгоритм_на_смету.docx`:

- строка работы: графы 2-7 берутся из ГЭСН и объёма ВОР;
- строки ресурсов: графа 5 = расход на единицу, графа 6 = коэффициент, графа 7 = графа 5 × графа 6 × объём работы;
- цена ресурса:
  - если есть текущая цена ФГИС ЦС — графа 10;
  - если текущей цены нет — графа 8 = базовая цена, графа 9 = индекс, графа 10 = графа 8 × графа 9;
  - если ресурса нет в ФГИС ЦС — нужен КАЦ;
- прямые = ОТ + ЭМ + М;
- ФОТ = ОТ + ОТм;
- НР = ФОТ × процент Приказа 812/пр;
- СП = ФОТ × процент Приказа 774/пр;
- итог позиции = прямые + НР + СП.

## Следующий хирургический шаг

1. Подключить `rim_lsr_trace_service.build_position_trace()` к существующему API отдельным необязательным endpoint, например:

   - `POST /api/lsr/rim-trace`

   Не менять контракт `POST /api/lsr/assemble`.

2. Добавить экспорт трассы в XLSX-шаблон Приложения 3.

   Важно: это должен быть renderer, а не новый расчётный движок. Расчёт уже должен приходить из trace rows.

3. Добить ФСЭМ-связку “машина → машинист”.

   Сейчас сервис честно работает с тем, что есть в норме. Если импорт ГЭСН уже даёт строки `machinist`, всё хорошо. Если в базе есть только машины, нужен отдельный справочник соответствий ФСЭМ:

   - код машины;
   - средний разряд машиниста;
   - код труда машиниста;
   - коэффициент / расход машиниста на машино-час.

4. Добавить golden-test на реальную позицию из GRAND-сметы.

   Не тащить большой XLSX в репо. Нужно вынести минимальную обезличенную fixture-позицию: код ГЭСН, объём, ресурсы, цены, ожидаемые строки.

## Риски

- ФСЭМ PDF в папке плохо извлекается текстом: символы дублируются. Не строить на нём прямой parser без отдельной проверки качества.
- НР/СП сейчас частично seed/catalog driven. Для промышленного нормоконтроля нужна полная таблица Приказов 812/пр и 774/пр с проверкой по сборнику/виду работ, а не только keyword fallback.
- Коэффициенты стеснённости применяются по виду ресурса. Выбор основания коэффициента пока должен приходить извне или из отдельного классификатора условий, а не угадываться LLM.
- РИМ-трасса не должна превращаться в жёсткий детерминированный “ответчик”. LLM может помогать находить контекст и объяснять, но числа и строки ЛСР должны оставаться проверяемым кодом.

## Правило для следующего агента

Не переписывать сметный модуль целиком. Сначала сохранить существующий контракт:

```text
ГЭСН -> ресурсы -> ФГИС/КАЦ цены -> РИМ trace -> renderer/export
```

Любое новое поведение добавлять маленькими слоями и закрывать отдельным тестом.
