# ЛЕС как мозговой центр проекта — архитектура и план

Версия 2 (2026-06-14), на основе исследования трёх областей: методологии «второго мозга»
(BASB/CODE, PARA, Zettelkasten, LYT/MOC, backlinks), архитектура knowledge-graph/GraphRAG, и
управление знаниями строительного проекта (ISO 19650 CDE, Uniclass/OmniClass, decision logs,
норм↔элемент, WBS/LBS/захватки). Источники — в конце.

## Позиционирование

**ЛЕС — не RAG-утилита, а мозговой центр проекта.** Организующая сущность — **объект строительства**;
вокруг него связаны нормативы, решения, BIM-элементы, задачи, объёмы, почта. Работает в **двух режимах**:

- **Режим проекта** — чат/ретрив/досье **сужены к объекту** (всё знание проекта в одном месте, навигируемо).
- **Обычный RAG** — без фильтра, поиск по всему (текущее поведение сохранено).

Это паттерн **pool**: одна база, тег `project_id` на каждом чанке/узле/ребре; по умолчанию — режим
активного проекта, тумблер «искать везде» снимает фильтр. Общие сущности (нормативы СП/ГОСТ) — `scope=global`,
ссылаются из многих проектов, не дублируются.

## Главный архитектурный принцип: граф-форма, но инвертированная стоимость

GraphRAG (Microsoft) даёт **форму** — узлы-сущности + типизированные рёбра + связь с чанками + локальный/
глобальный поиск. Но его стоимость почти вся в LLM (извлечение сущностей/связей + community-суммаризация).
Мы берём форму и **переворачиваем модель затрат под ADR-11**:

> **Связи выводятся детерминированно (ID-джойны → regex/словарь → FTS/NER → малая модель → локальная LLM →
> облако), LLM — последняя ступень и только для того, что доказуемо не берётся правилами.**

Это ровно философия уже сделанного: **W11.2** (план/факт ВОР↔журнал по ID-джойну, 0 LLM), **W2.7** (словаря
хватило, no-go LLM-ступени), **W3.3** (локаль→облако P0/P1/P2). Граф — то же самое, применённое к связям.

Инвариант: **у каждого ребра — `extraction_method` + `confidence_tier` + provenance (source_chunk/record).
Нет анонимных рёбер.** Детерминированные рёбра доверенные сразу; LLM-предложенные входят как `candidate` и
промотируются в `trusted` только после детерминированной проверки заземления (сущности есть в источнике,
связь поддержана контекстом). Так «уверенная галлюцинация LLM» не отравляет доверенный граф.

## Как ложится на существующую архитектуру ЛЕС

| Слой графа | Существующее в ЛЕС | Что добавляем |
|---|---|---|
| Хаб-объект | — (датасеты ≠ стройки) | `les_projects` + `les_project_links` (тег-таблица) |
| Чанки/документы | Qdrant `les_rag_qwen3_06b`, датасеты | payload `project_id` + `scope`; состояние CDE на документах |
| Нормативы | НТД-RAG, `/api/rag/graph/edges` (FTS СП/ГОСТ) | `NormClause` (до пункта), рёбра requires/constrains |
| BIM | `cad_bim_elements/relations` (source_id, этаж, категория) | классификационный хребет (Uniclass-style коды как свойства) |
| Решения/заметки | `les_notes` (плоские) | `DecisionRecord` (структура + типизированные рёбра) |
| Задачи | `les_tasks` | рёбра concerns→Element / at→Захватка |
| Объёмы | `les_field_entries` (захватка/позиция/объём) | Захватка как LBS-хаб, %готовности |
| Граф-вьювер | `qdrant_visualizer` (W5.7) | наложение личного слоя + типизированные рёбра, фильтр по проекту |
| Связи | — | `les_edges` (типизированные, provenance) |
| Диспетчер | `chat.py` (детерм. каналы до admission) | project-scope фильтр + новые детерм. каналы |

Новые таблицы (`les_projects`, `les_project_links`, `les_edges`) — в метабазе рядом с `les_notes/tasks/field`.

## Онтология (намеренно маленькая, типизированная, доменная)

Сущности и ключевые рёбра (всё — из консолидированной модели строительного проекта):

| Сущность | Типизированные рёбра | Источник |
|---|---|---|
| **Project (объект)** | хаб; `part_of_project` от всего | новая |
| **Container (документ/чертёж/модель)** | `supersedes` (ревизии), состояние WIP/Shared/Published/Archived | НТД/CAD-импорты |
| **NormClause (пункт норматива)** | `requires_property`→(категория,ключ), `constrains`→категория{op,val} | НТД-RAG (до пункта) |
| **Element (BIM)** | `realized_by`→System, `hosted_in`→Space, `located_on`→Floor | cad_bim_elements |
| **System / Space / Floor** | `contains`, `composed_of` (классификационный хребет) | категории/этажи BIM |
| **Захватка (зона, LBS)** | `part_of`→Floor; хаб work×location | les_field_entries |
| **BoQItem / WorkPackage (ВОР)** | `performed_at`→Захватка, `covers`→Element | ВОР (W11) |
| **FieldVolumeEntry** | `reports`→BoQItem @Захватка (→ %готов) | les_field_entries |
| **DecisionRecord / Task** | `concerns`→Element, `references`→Container, `at`→Захватка, `justified_by`→NormClause, `supersedes` | les_notes/les_tasks |

Четыре семейства рёбер делают это «мозгом», а не БД: **состояние/ревизия** (аудит, ISO 19650),
**классификационный хребет** (навигация element→system→space→floor), **трассируемость решение↔артефакт↔
норматив** (обоснование — самое ценное семейство), **стык work×location×element на захватке** (реальность,
%готовности = Σфакт/Σплан, считается без LLM).

## Извлечение рёбер — детерминированная лестница (0 LLM в ядре)

1. **ID-джойны** (высшая достоверность, по умолчанию): ВОР-строка ↔ ключ журнала объёмов (**уже есть в W11.2**),
   `source_id` ↔ элемент, `project_id`-привязка. Точные, аудируемые, без NLP.
2. **Regex/словарь**: коды НТД (СП/ГОСТ — **переиспользуем `/api/rag/graph/edges` FTS**), `[[вики-ссылки]]`,
   захватка/позиция, Source ID (**переиспользуем `cad_bim_highlight`**).
3. **NER/FTS-ко-встречаемость**: слабые кандидаты-рёбра.
4. **Малая модель → локальная LLM → облако**: только для неявных связей и парсинга свободного текста пункта
   норматива в предикат `{property, op, value}` — gated, как `candidate`, с проверкой заземления.

Каждое ребро тегируется методом и достоверностью; детерминированные — доверенные на запись, LLM — в карантин.

## Двойной режим ретрива (pool + project_id)

- Один Qdrant + один граф; payload `project_id` на каждом чанке, узле, ребре.
- **По умолчанию — фильтр по активному проекту**; тумблер «искать везде» снимает фильтр (bleed — opt-in, не
  случайный). Фильтры project-scope и метаданные **композируются**.
- Общие сущности (нормативы, организации, материалы) — `scope=global`, ссылаются, не копируются.
- Ложится на `chat.py`: project-scope добавляется в фильтр ретрива; «обычный RAG» = старый путь без фильтра.

## Инкрементальность и стоимость (write-time vs backfill)

- **Write-time (синхронно, только детерминированно):** ID-джойны, regex/коды, словарь, ко-встречаемость —
  дёшево, без LLM, граф сразу полезен для локальных запросов.
- **Backfill (батч/офф-пик, gated):** LLM-обогащение (неявные связи, парсинг пунктов), entity-resolution,
  глобальные суммаризации. Уважает рантайм-правило «не жечь compute зря»: тяжёлое — батчем, по расписанию.

## Практики «второго мозга», адаптированные (не строим generic Obsidian)

- **Организация по жизненному циклу, не по теме** (PARA): корзины Проект/Архив/Ресурс **вычисляются из статусов**
  (открытая задача→Проект, закрытая→Архив, норматив→Ресурс), не ручная папка.
- **Атомарность + стабильный ID** (Zettelkasten): неизменные ID, связи — типизированные рёбра между ID, не
  ломаются при переименовании. «Литературный слой» (документы) ингестится автоматически; человек авторит синтез.
- **Хабы = вычисляемые запросы + тонкая курация** (LYT/MOC): «КАРТА ОБЪЕКТА» — это query-backed досье, не
  ручная папка; человеку остаётся только аннотация/порядок («это головной СП»).
- **Бэклинки типизированные и сгруппированные** (не плоский счётчик): узел с 200 бэклинками полезен, когда 3 из
  них `supersedes`, а 197 `mentioned-in` — показываем по типу ребра.
- **Метрика — извлечение/использование, не захват** (collector's fallacy): «захвачено, но ни разу не всплыло» —
  видимый долг, не заслуга. Recall по использованию **уже есть (W16.1)**.
- **Дистилляция по требованию, оригинал всегда сохранён**: не суммируем на ингесте; слой «суть» вычисляем
  (управляющие числа, связывающий пункт, изменённое количество); LLM-сводка — только по запросу.

Мета-вывод: generic-инструменты сваливают это на ручную дисциплину человека (где все и проваливаются —
пропускают Distill, не пишут permanent notes, over-link). Преимущество ЛЕС — он **знает статусы, структуру
источников и семантику связей**, поэтому **вычисляет слои, которые человек пропускает** (привязки, связи, «суть»,
типы рёбер), а человеку оставляет одно несводимое — **синтез/решение**.

## Карты — Волна 17 (с учётом архитектуры)

> Сквозные инварианты для всех карт: provenance на каждом ребре; `confidence_tier`; детерминированное=доверенное /
> LLM=candidate-в-карантине; project_id default-on. Приёмка каждой — детерминированно (0 LLM), кроме явно `[gate]`.

- **W17.1 Сущность объекта + двойной режим** · M — `les_projects` + `les_project_links`; payload `project_id`/
  `scope` на чанках Qdrant; фильтр project-scope в `chat.py` (default — активный проект, тумблер «везде»);
  чат-команда «проект: …»/селектор. Нормативы — `scope=global`, достижимы в обоих режимах.
  **Приёмка:** вопрос в режиме проекта A → документы A; тумблер → всё; норматив достижим всегда. (фундамент)
- **W17.2 Хранилище типизированных рёбер + детерм. вывод** · L — `les_edges` (src/dst kind+id, edge_type,
  method, confidence, provenance); вывод переиспользует: НТД (graph/edges FTS), `[[вики]]` (regex), BIM source_id
  (`cad_bim_highlight`), ВОР↔журнал (W11.2 джойн), захватка. Write-time + backfill-джоб.
  **Приёмка:** заметка «[[СП 4.13130]] по захв. Б2 для <source_id>» → 3 типизированных ребра с provenance, 0 LLM.
- **W17.3 Доменная онтология: классификационный хребет + состояния** · L — ✅ 2026-06-14
  (`proxy/services/ontology_service.py`, 0 LLM). Хребет Floor→System→Category поверх `cad_bim_elements`
  (система: системное свойство → словарь категории → «Прочее»), обход `elements_in`; состояния Container CDE
  (`les_containers`, конечный автомат WIP→Shared→Published→Archived, `supersede`→ребро в `les_edges`); Захватка-хаб
  LBS из журнала объёмов. API `/api/ontology/*`, интеграция в досье. 12 тестов. **Приёмка пройдена:** обход «элементы
  системы вентиляции на этаже 3» ✅; «решение на published или WIP чертеже?» через `get_container().cde_state` ✅.
- **W17.4 Слой решений (память проекта)** · M — ✅ 2026-06-14 (`proxy/services/decision_service.py`, 0 LLM).
  `les_decisions` (вопрос/решение/обоснование/статус/теги, партиционирована по объекту) + веер рёбер
  `justified_by`/`concerns`/`at`/`references`/`supersedes` (regex-экстракторы W17.2); бэклинки сгруппированы по типу
  ребра. Чат-команды «реши: …»/«решения», API `/api/decisions/*`, в досье. 8 тестов. **Приёмка пройдена:** «реши:
  балку до 400 мм, обоснование по СП 20.13330 …Source ID: beam-12 …[[корпус Б]]» → justified_by→СП, concerns→beam-12,
  references→корпус Б, at→захватка.
- **W17.5 КАРТА ОБЪЕКТА (досье) + вычисляемый MOC** · M `[gui]` — `GET /api/project/{id}/dossier` (нормативы в
  области, решения, открытые задачи, %объёмов по захваткам, сводка BIM по этажам, почта — SQL+сервисы, 0 LLM);
  экран «КАРТА ОБЪЕКТА»; PARA-корзины из статусов. **Приёмка:** реальный объект → полный паспорт одним экраном.
- **W17.6 Граф проекта** · M — наложение личного слоя + типизированных рёбер на W5.7, фильтр по объекту,
  бэклинки по типу, хаб=объект. **Приёмка:** объект-хаб с решениями/задачами/НТД/элементами вокруг.
- **W17.7 Норматив→элемент: контроль полноты/соответствия** · M `[gate:dict→llm]` — `NormClause`
  `requires_property`/`constrains`; проверка полноты («у каждой несущей стены — fire-rating по СП X»). Сопоставление
  пункт↔категория — сперва словарь/категории; LLM-парсинг свободного текста пункта — только где правила бессильны,
  candidate+заземление. **Приёмка:** список элементов без обязательного свойства по пункту.
- **W17.8 LLM-синтез (gated, последняя ступень)** · S `[gate]` — по требованию: «суммируй решения по объекту»,
  «что не закрыто перед сдачей», глобальные community-сводки. Только после доказанного детерм. слоя; candidate-рёбра
  в карантине. **Приёмка:** сводка с трассировкой к источникам; ни одного недоказанного факта.

### Порядок и зависимости
```
W17.1 (объект+режим) ──► W17.2 (рёбра) ──► W17.3 (онтология) ──► W17.5 (досье) ──► W17.6 (граф)
                                   └──► W17.4 (решения) ──┘
W17.3 ──► W17.7 (норм↔элемент) ──► W17.8 (LLM-синтез, last resort)
```
W17.1 разблокирует всё. W17.2 — спина. W17.5 — первый видимый payload оператору.

## Риски (назвать прямо)
- **Не строить generic Obsidian.** Организующий принцип — объект строительства; рёбра типизированные и доменные,
  не «всё связано со всем». Сила ЛЕС — в домене.
- **LLM только last-resort.** Ядро 0 LLM; LLM-рёбра в карантине до заземления, иначе тихо отравят ретрив.
- **project_id default-on**, иначе межпроектная утечка. «Искать везде» — явный opt-in.
- **Граф не для всего.** Локальные «найди чанк» — обычный вектор-RAG; граф — для многошаговых, агрегатных,
  связных и аудит-запросов. Не навешивать граф там, где хватает вектора.
