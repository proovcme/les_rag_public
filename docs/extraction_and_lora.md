# Извлечение по схеме, чтение сканов и LoRA — что, зачем, как

Научно-популярно и **воспроизводимо** про три вещи на пути ЛЕСа к разбору
строительной документации:

1. **Извлечение по схеме** — «украли» приём и за вечер встроили;
2. **Чтение скана, который не берёт OCR** — взломали на реальных чек-листах;
3. **LoRA** — сознательно **не** торопимся, и в конце объясняем почему.

Текст с образцами кода: если кто-то захочет так же — сможет повторить у себя.
Всё проверено на реальных сканах объекта (чек-листы МФЗ Б4 Э23 L5), цифры —
оттуда.

Сквозная идея — **принцип LLM-минимализма**: большая модель это *последний*
инструмент, а не первый. Сначала алгоритм, потом словарь, потом маленькая
модель, и лишь потом — большая LLM, в самом конце — облако. Всё ниже про то,
чтобы LLM работала **меньше и надёжнее**, а не больше.

---

## Часть I. «Кража»: извлечение по схеме

### Что это

Два способа спросить машину про документ:

- **«расскажи»** — пересказ, каждый раз разный, в таблицу не загонишь;
- **«заполни анкету»** — даёшь жёсткий бланк (JSON-схему), получаешь ровно эти
  поля в одинаковом виде.

**Извлечение по схеме** — это второй способ. Идею подсмотрели у проекта
[`lift`](https://github.com/datalab-to/lift) (команда Marker/Surya): даёшь
JSON-схему, получаешь валидный по ней JSON. Мы взяли **приём, а не продукт** —
без их несвободной лицензии весов, без torch+CUDA (у нас Apple-MLX), без
философии «VLM на каждое поле».

### Зачем

Наивное «дай мне JSON» **не работает на серьёзных документах**: модель придумает
лишние поля, сломает формат, **сочинит числа**. Для сметы выдуманное число — брак.
Схема превращает пожелание в **контракт**: гарантированная форма + проверяемость
(число обязано быть числом, единица — из списка). А по минимализму — модель тут
только на «мутном» участке, чистые таблицы идут мимо неё, через `table_query`.

### Как (с кодом)

Механика — пять шагов, работает на **любом** движке: схема → промпт → вызов →
парсинг → валидация, и если невалидно — **«проверь-и-почини»** (переспросить с
ошибками). Минимальная самодостаточная реализация:

```python
import json, re

def parse_json(text: str):
    """Достать JSON из ответа модели (с ```-ограждением, в прозе, как угодно)."""
    if not text:
        return None
    m = re.search(r"\{.*\}|\[.*\]", text, re.DOTALL)
    if not m:
        return None
    try:
        return json.loads(m.group(0))
    except Exception:
        return None

def validate(instance, schema) -> list[str]:
    """jsonschema если есть, иначе мини-проверка (type/required/enum)."""
    try:
        import jsonschema
        return [e.message for e in jsonschema.Draft7Validator(schema).iter_errors(instance)]
    except ImportError:
        errs = []
        if schema.get("type") == "object" and isinstance(instance, dict):
            for k in schema.get("required", []):
                if k not in instance:
                    errs.append(f"нет обязательного поля '{k}'")
        return errs

def extract(schema, instruction, context, call_llm, max_attempts=3):
    """call_llm(prompt) -> str. Возвращает валидный по схеме объект или None."""
    prompt = (
        f"{instruction}\nИзвлеки СТРОГО по JSON-схеме, ответь только JSON:\n"
        f"{json.dumps(schema, ensure_ascii=False)}\n\nДокумент:\n{context}"
    )
    for _ in range(max_attempts):
        reply = call_llm(prompt)
        data = parse_json(reply)
        errors = ["не JSON"] if data is None else validate(data, schema)
        if not errors:
            return data
        prompt = (  # шаг «починки»: возвращаем модели её же ошибки
            f"Ответ не прошёл схему: {errors}\nПредыдущий ответ:\n{reply}\n"
            f"Исправь, верни только валидный JSON по схеме:\n{json.dumps(schema, ensure_ascii=False)}"
        )
    return None
```

Облачные провайдеры умеют держать схему сами (нативный `response_format`
json_schema) — тогда `call_llm` просто передаёт его в тело запроса. Локальная
модель — через цикл «починки» выше. **Один код, оба движка.**

В коде ЛЕСа это: `proxy/services/structured_extract.py` (движок),
`proxy/services/extract_service.py` (привязка к провайдеру),
`POST /api/extract/structured` (точка входа).

#### Урок №1: описания полей сильнее смены модели

Первый прогон на реальном чек-листе розеток дал кашу: поле `position` (артикул)
заполнилось **номерами строк** 1, 2, 3. Соблазн — «модель слаба, нужна побольше».
Вместо этого — **добавили в схему описания полей**, одну фразу на поле. Та же
модель, тот же текст:

```python
# Было: {"position": {"type": "string"}}
# Стало:
{"position": {"type": "string",
              "description": "артикул, 4-значное число (5508), НЕ порядковый номер строки"}}
```

| Поле       | Без описаний              | С описаниями                       |
|------------|---------------------------|------------------------------------|
| `mark`     | «…&Б 15» (мусор)          | **`+16.800`** ✓                    |
| `position` | `1, 2, 3` (номер строки)  | **`5508, 51601, 5153`** (артикулы) |

**Дешёвый рычаг обошёл большой молоток.** Запомните этот урок — он повторится.

Чего схема **не** починила: рукописные колонки. Их OCR теряет **полностью** — в
тексте этих данных нет. Хуже того, на части сканов OCR не взял вообще ничего. Это
ведёт ко второй части.

## Часть II. Прочитать скан: OCR против vision

### Что это

Извлечению нужен **текст**, а скан — картинка. Шаг «прочитать» имеет два пути:

- **OCR** (у нас Tesseract): картинка → текст по буквам. Дёшево, но спотыкается
  на плохом скане, печатях, рукописи.
- **Vision-модель** (VLM, у нас локальный **Qwen3-VL** через ollama): смотрит на
  картинку целиком, как человек, и сразу выдаёт структуру — включая рукопись.

OCR-шаг тривиален:

```python
import subprocess, tempfile, os
import fitz  # pymupdf

def ocr_page(pdf_path, page=0, dpi=200, lang="rus"):
    pix = fitz.open(pdf_path)[page].get_pixmap(dpi=dpi)
    png = os.path.join(tempfile.mkdtemp(), "p.png"); pix.save(png)
    return subprocess.run(["tesseract", png, "stdout", "-l", lang, "--psm", "6"],
                          capture_output=True, text=True, timeout=120).stdout
```

### Зачем

Потому что на реальных чек-листах объекта OCR показал потолок. Из трёх:

- один (розетки) OCR прочитал прилично — печатный реестр вышел;
- **два (светильники, щиты) OCR превратил в мусор** — бессвязные обрывки вроде
  «Этаж итовог», ни одной пригодной строки.

То есть **на 2 из 3 реальных сканов текстовый путь — ноль**. Плюс рукописные
колонки приёмки нигде. Vision не роскошь — необходимость.

### Как (с кодом и граблями)

Дорога заняла вечер. Уроки стоит записать:

1. **Не тот рантайм.** `mlx-vlm` (нативный для Apple-MLX) Qwen3-VL пока не тянет —
   выдаёт пустоту. Зато **ollama** держит.
2. **Не тот эндпоинт.** Через ollama `/api/generate` ответ уходил «в никуда»;
   правильный для картинок — **`/api/chat`** (ответ в `message.content`).
3. **Главный враг — «режим размышления».** Qwen3-VL «думает вслух»: на сложном
   скане уходит в бесконечное рассуждение и **до ответа не доходит**. Тумблеры
   `think:false` и `/no_think` в нашей версии ollama его **не гасят**.

Развязка — два хода. **Ход 1: Modelfile** с не-думающим обработчиком
(`qwen3-vl-instruct` вместо `qwen3-vl-thinking`):

```dockerfile
# qwen3-vl-nt.Modelfile
FROM qwen3-vl:4b
RENDERER qwen3-vl-instruct
PARSER qwen3-vl-instruct
PARAMETER temperature 0
```

```bash
ollama create qwen3-vl-nt -f qwen3-vl-nt.Modelfile
```

Этого мало — модель всё равно генерит `<think>…`. **Ход 2: при вызове подсунуть
пустой, "уже подуманный" блок** последним assistant-сообщением — и она сразу
переходит к ответу:

```python
import base64, json, urllib.request

def vision_extract(png_path, schema, model="qwen3-vl-nt"):
    img = base64.b64encode(open(png_path, "rb").read()).decode()
    prompt = (
        "Скан строительного чек-листа — таблица оборудования по помещениям. "
        "Извлеки ВСЕ строки в JSON-массив СТРОГО по схеме (соблюдай назначение "
        "полей из их description). Включи рукописные значения. Только JSON.\n\n"
        "Схема:\n" + json.dumps(schema, ensure_ascii=False)
    )
    body = {
        "model": model,
        "messages": [
            {"role": "user", "content": prompt, "images": [img]},
            # ↓ ключевой трюк: пустой закрытый <think> гасит «размышление»
            {"role": "assistant", "content": "<think>\n\n</think>\n\n"},
        ],
        "stream": False,
        "options": {"temperature": 0, "num_predict": 2200},
    }
    req = urllib.request.Request(
        "http://127.0.0.1:11434/api/chat",
        data=json.dumps(body).encode(),
        headers={"content-type": "application/json"},
    )
    resp = json.loads(urllib.request.urlopen(req, timeout=400).read())
    return parse_json(resp["message"]["content"])  # parse_json из Части I
```

Результат — модель завершает **штатно** (`done: stop`) и отдаёт чистый JSON.

И снова — **тот же урок №1**: описания полей. Без них база путала графы (марку
щита клала в «помещение»). Та же схема с описаниями:

```python
SCHEMA = {"type": "array", "items": {"type": "object", "properties": {
    "помещение":         {"type": "string", "description": "номер помещения (451 или 5)"},
    "оборудование":      {"type": "string", "description": "марка/обозначение: ШР5, РШТ5.1, ШС65, 'Розетка одинарная' — НЕ номер помещения"},
    "позиция":           {"type": "string", "description": "артикул 4-значный (5508), если указан"},
    "количество":        {"type": "string", "description": "количество, часто рукописное"},
    "отметка_о_приемке": {"type": "string", "description": "рукописная отметка: 'не тесн.', 'удержано', подпись+дата"},
}}}
```

#### Число, ради которого всё затевалось

Чек-лист **щитов**, где OCR дал круглый ноль:

| Путь | Результат на странице щитов |
|------|------------------------------|
| **OCR (Tesseract)** | **0** пригодных строк (сплошной мусор) |
| **Vision + описания** | **15 строк, маппинг верный (оборудование 15/15), 4 с рукописной приёмкой** |

А на **розетках** (где OCR брал только печатное) vision с описаниями вытащил
**17/17 строк со ВСЕМ рукописным**:

```json
{"помещение": "451", "оборудование": "Блок настенных розеток",
 "позиция": "5602", "количество": "3", "отметка_о_приемке": "ГЛ 27.04 ГН 27.04"}
```

`количество: "3"` и `отметка: "ГЛ 27.04 ГН 27.04"` (инициалы инспекторов + даты) —
**рукописные**, которых у OCR нет в принципе. **Vision вытащил структуру и рукопись
там, где OCR — ноль. Данными, не на словах.**

## Часть III. LoRA

### Что это

Большая модель — миллиарды настроенных «ручек». Дообучить все под свою задачу —
долго и дорого, как второе высшее. **LoRA (Low-Rank Adaptation)** — это «курсы
повышения квалификации»: тонкая **съёмная насадка** (десятки МБ, обучается часами
на ноутбуке) поверх большой модели. Модель помнит всё, но с уклоном в вашу тему.

### Зачем — и почему мы её НЕ сделали

Соблазн: «дообучим локалку под стройку». Но LoRA — последняя ступень, и вот
почему мы до неё **не дошли**:

- **Против минимализма.** Числа/таблицы дообучать бессмысленно — там детермини-
  рованный `table_query` точнее. LoRA — только на «мутном» хвосте.
- **Гонка моделей.** Насадка привязана к базе. Вышла новая Qwen — переобучай.
- **Главный аргумент «за» — снят.** Думали: «рукопись теряется → нужна LoRA».
  Оказалось — **базовый Qwen3-VL рукопись уже читает** (количества, подписи,
  даты), стоило лишь описать поля. Зона для LoRA схлопнулась до **символьной
  шумности** распознавания («неподрежен» вместо «не подрессорен»). LoRA *могла бы*
  это подполировать — но данные уже извлечены и структурированы.

### Как (если когда-нибудь дойдём)

- **База — наш же `qwen3-vl-nt`** (уже работает).
- **Данные = eval-набор.** Разметка «скан → правильный JSON» нужна, чтобы
  *измерить*. Та же разметка — **обучающая выборка** для LoRA. Размечаешь раз,
  используешь дважды.
- **Тренировка — `mlx_lm.lora` на M4**, адаптер мелкий, лицензия чистая (наш
  адаптер на весах Qwen, не на весах `lift`).
- **Метрика — field-accuracy** (ниже), петля оценки готова.

## Лестница решений

```
Задача: достать структуру из документа
│
├─ Чисто и в таблице?         → детерминированный парс (table_query). LLM не нужна.
│
└─ Мутно / скан / текст?      → извлечение по схеме (Часть I)
      │
      ├─ Текст читается?       → OCR + LLM + ОПИСАНИЯ полей.            ← доказано
      │
      ├─ OCR = мусор/рукопись? → Qwen3-VL (qwen3-vl-nt) + ОПИСАНИЯ.    ← ДОКАЗАНО:
      │                          15 строк против 0 у OCR, рукопись читается
      │
      └─ Vision вышел на плато, и обязательно локально?  → ТОЛЬКО ТОГДА LoRA
```

Каждая ступень дороже предыдущей. Идём снизу вверх, останавливаемся на первой,
что закрывает задачу. На сегодня закрыты первые три — **до LoRA дело не дошло**.

## Как измерять (field-accuracy, как у lift)

Чтобы решать «нужна ли LoRA» арифметикой, а не на глаз — меряем долю верно
извлечённых полей против эталона:

```python
def flatten(obj, prefix=""):
    """JSON → {путь_к_листу: значение}."""
    out = {}
    if isinstance(obj, dict):
        for k, v in obj.items():
            out.update(flatten(v, f"{prefix}/{k}" if prefix else k))
    elif isinstance(obj, list):
        for i, v in enumerate(obj):
            out.update(flatten(v, f"{prefix}[{i}]"))
    else:
        out[prefix] = obj
    return out

def field_accuracy(expected, actual):
    exp, act = flatten(expected), flatten(actual)
    matched = sum(1 for k, v in exp.items() if str(act.get(k, "")).strip() == str(v).strip())
    return matched / len(exp) if exp else 0.0
```

В ЛЕСе это `tools/extract_bench.py` + набор кейсов `golden/extract_eval_set.json`.

## Повторить у себя (полный рецепт)

```bash
# 1. Зависимости: ollama + python-либы
#    ollama: https://ollama.com  (модель qwen3-vl нужно один раз получить)
ollama pull qwen3-vl:4b            # если реестр доступен из вашей сети
pip install pymupdf jsonschema     # OCR-обвязка + валидатор; tesseract — системный пакет

# 2. Не-думающий vision-профиль (Modelfile из Части II)
ollama create qwen3-vl-nt -f qwen3-vl-nt.Modelfile

# 3. В Python: ocr_page() ИЛИ vision_extract() из статьи
#    - печатный скан  → ocr_page() → extract(SCHEMA, ...)
#    - мусорный/рукопись → vision_extract(png, SCHEMA)  (с описаниями полей!)
```

Три правила, выстраданные на граблях:
1. **Описывайте поля** в схеме — это сильнее, чем брать модель побольше.
2. Для vision в ollama — **`/api/chat`**, не `/api/generate`.
3. Думающую модель гасите **`qwen3-vl-instruct` + prefill пустого `<think>`**.

## Итог в четырёх строчках

1. **Украли** дешёвый приём — извлечение по схеме — и встроили; **описания полей**
   оказались сильнее смены модели.
2. **Взломали чтение** мусорных сканов: локальный Qwen3-VL (`qwen3-vl-nt`) достал
   **15 строк с рукописной приёмкой** там, где OCR дал **ноль**. Данными, не верой.
3. **LoRA не сделали** — главный аргумент («рукопись теряется») снят: база её
   читает. Осталась лишь шлифовка хвоста.
4. Решение «нужна ли LoRA» стало **арифметикой**: дособрать eval-набор, прогнать
   `extract_bench`, посмотреть на число — и только оно скажет «да» или «нет».
