# Алгоритм: как делается смета в ЛЕС (единый поток — для оператора)

Этот документ **сшивает** разрозненные сметные ALGO-доки в один связный поток «от объёма работ
до общей цены». Детали каждого кирпича — в своём доке (ссылки по тексту); здесь — как они стыкуются
и **как этим пользоваться**. Числа считает КОД (ADR-11, 0 LLM в расчёте); каждое число цитируемо и
проверяемо тестом. Эталон сквозной проверки — позиция **«Устройство обрешётки» (ГЭСН 12-01-034-02)**,
которая воспроизводится бит-в-бит: **Всего по позиции = 11813.04 ₽** при объёме 0.61.

## 1. Поток end-to-end (с формулами)

```
ВОР (объёмы работ)                                    ← что и сколько делаем
  │  каждая работа → код нормы ГЭСН
  ▼
норма ГЭСН (gesn_service, база 42408 норм из ФГИС ЦС) ← расход на единицу
  │  расход × объём → строки ресурсов (труд / машины / материалы)
  ▼
цены ресурсов
  │  есть код в «Сплит-форме» ФГИС ЦС (локально)      → fgis_price.lookup   [[ALGO-fgis-price]]
  │  нет кода (материал)                              → КАЦ (≥3 КП)         [[ALGO-kac]]
  ▼
агрегация по видам:  ОЗП=Σтруд · ЗПМ=Σмашинисты · машины · М=Σматериалы
  │  ЭМ = машины + ЗПМ
  ▼
стеснённость (k к ОЗП и ЭМ; материалы не трогаются)  [[ALGO-stesnennost]]
  ▼
прямые = ОЗП + ЭМ + М                ФОТ = ОЗП + ЗПМ
  │
  ▼
НР / СП по виду работ (nr_sp, Приказы 812/пр · 774/пр; оба % от ФОТ)
  │  НР = ФОТ · нр%      СП = ФОТ · сп%
  ▼
Всего по позиции = прямые + НР + СП    (lsr_assembly.compute_position)  [[ALGO-lsr-assembly]]
  │  свод позиций по разделам → lsr_assembly.assemble
  ▼
ИТОГО СМР = Σ(прямые + НР + СП)        ← объектная смета: object_estimate  [[ALGO-object-estimate]]
  + непредвиденные 2%
  + НДС 20%
  ▼
ВСЕГО (общая цена с НДС)
```

**Бухгалтерия позиции** (выверена на эталоне, обрешётка, объём 0.61):

| Величина | Формула | Эталон |
|---|---|---|
| ОЗП | Σ labor | 3750.23 |
| ЗПМ | Σ machinist (ОТм) | 458.68 |
| ЭМ | машины + ЗПМ | 992.40 |
| М | Σ material | 83.62 |
| Прямые | ОЗП + ЭМ + М | 4826.25 |
| ФОТ | ОЗП + ЗПМ | 4208.91 |
| НР | ФОТ × нр% (109, «Кровли») | 4587.71 |
| СП | ФОТ × сп% (57, «Кровли») | 2399.08 |
| **Всего** | Прямые + НР + СП | **11813.04** |

Граница ответственности: **позицию** (до «Всего по позиции» + свод по разделам) считает
`lsr_assembly`; **хвост объектной сметы** (СМР → непредвиденные → НДС → ВСЕГО) добавляет
`object_estimate_service.estimate` (`smr → contingency 2% → before_vat → vat 20% → grand_total`).

## 1а. Доказательная трасса и форма ЛСР (Приложение № 4)

Поверх расчёта строится **РИМ-трасса** (`rim_lsr_trace_service`) — те же числа, разложенные по графам
2-12 формы ЛСР с происхождением каждой цены (текущая ФГИС ЦС / база×индекс / КАЦ / явная / `missing`).
Это не второй калькулятор: `build_position_trace` берёт норму и цены тем же путём, что `lsr_assembly`,
и раскладывает одну позицию в проверяемые строки (работа → ОТ(ЗТ)/ЭМ/ОТм/М → прямые/ФОТ/НР/СП/Всего).

Трасса рендерится в XLSX по форме **Приложения № 4 к Методике 421/пр** (форма ГРАНД-Сметы,
`rim_trace_xlsx_service`); числа НЕ пересчитываются — рендер берёт ГОТОВУЮ трассу:

```
одна позиция         → build_position_trace → render_trace_xlsx  → POST /api/lsr/rim-trace/export
многопозиц. смета     → build_lsr_trace      → render_lsr_xlsx    → POST /api/lsr/lsr-trace/export
                         │ позиции по разделам (поле section), непрерывная нумерация,
                         ▼ «Итого по разделу N» + общий свод «ВСЕГО по смете»
```

Шапка формы — стройка/объект/ЛСР №/ресурсный метод/уровень цен/субъект + разбивка сметной стоимости
(ОЗП/ЗПМ/нормативные чел.-ч рабочих и машинистов). Свод многопозиционной = Σ позиций (код, не LLM).
Gold сохраняется: позиция 11813.04, смета 2×позиции = 23626.08. ЛСР = **Приложение № 4** (Прил.3 —
объектный расчёт). Индекс для `база×индекс` — из ФГИС ЦС, официально из писем Минстроя ИФ/09
([[minstroy-indices-source]], бэклог). Детали графов и свода — [[ALGO-lsr-assembly]].

## 2. Тариф труда (почему ОЗП ≠ 0)

В базе ФГИС ЦС трудозатраты приходят как **«Средний разряд работы N,M» без кода ресурса** — а без
кода цена не находится, ОЗП был бы 0. ЛЕС закрывает это **read-time**: из наименования выводится
тарифный код `1-100-NM` (`gesn_service._labor_tariff_code`, regex по «разряд N,M»). Эталонный разряд
**2,5 → код `1-100-25`**, цена которого есть в «Сплит-форме» → ОЗП базовых норм ≠ 0.

Граница: **машинист-агрегат** «Затраты труда машинистов» идёт **без разряда** → код не выводим →
строка помечается флагом «нет цены» (а не молча 0). В семени эталона ОТм заданы явной ценой (тариф
по разряду), поэтому ЗПМ эталона точен.

## 3. Как пользоваться (чат-каналы Совушки)

Все функции доступны в чате одной строкой (детерминированные каналы `smeta_chat_service`, 0 LLM,
до RAG). Каналы и примеры:

**Цена ресурса по коду** — «цена `<код>`», «сколько стоит `<код>`»:
```
> цена 91.05.01-017
Краны башенные, г/п 8 т · маш-ч
Сметная цена (текущая): 865.44 руб. (базовая 622.62 × индекс 1.39)
Санкт-Петербург · 2 кв. 2025 · книга …
```
(Сходится с проверенным эталоном кровли до копейки: краны 622.62×1.39=865.44.)

**Нужен ли КАЦ** — «нужен ли КАЦ `<код>`», «есть в ФГИС `<код>`»:
```
> нужен ли кац 01.7.99.99-9999
Кода 01.7.99.99-9999 НЕТ в ФГИС ЦС — нужен КАЦ: ≥3 коммерческих предложения на материал,
выбрать экономичный, цену учесть в смете (Инструменты → КАЦ).
```
Логика: `kac_service.needs_kac` смотрит, **есть ли код в книге цен** ФГИС ЦС (локаль-первый вердикт).

**Коэффициент стеснённости** — «коэффициент стеснённости в городе»:
```
> коэффициент стеснённости город
Коэффициент стеснённости — «<условие>»: к ОЗП ×1.15, к ЭМ ×1.15. …
Применяется к ОЗП и ЭМ позиции (материалы не трогаются).
```
Каталог условий — `config/domain/stesnennost.yaml` (значения типовые, правятся в yaml).

**Сборка позиции от кода ГЭСН** — «собери `<код>` объём N»:
```
> собери ГЭСН12-01-034-02 объём 0.61
Устройство обрешётки с прозорами из брусков · ГЭСН12-01-034-02 · объём 0.61 100 м2
ОЗП 3 750.23 + ЭМ 992.40 + М 83.62 = прямые 4 826.25
ФОТ 4 208.91 → НР 4 587.71 (109%) + СП 2 399.08 (57%) · вид работ: Кровли
ИТОГО по позиции: 11 813.04 руб.
```
Это сквозной **эталон**: код без ресурсов → движок разворачивает норму → цены → НР/СП → 11813.04.
Если норма не в локальной базе — канал добирает её из ФГИС ЦС (бесплатно) или smetnoedelo (с токеном).

**Объектная смета из фразы** — «дай смету на `<объект>`» (`object_estimate`):
```
> дай смету на деревянный двухэтажный дом 100 м²
Смета: Деревянный дом (брус), типовой ИЖС · 100 м², 2 эт. (укрупнённо, по типовому составу работ)
• Устройство обрешётки · 12-01-034-02 · 0.6 100 м2 — … ₽
• …(7 позиций конструктива: фундамент, стены из бруса, перекрытия, обрешётка, кровля, полы, утепление)
  ИТОГО СМР (прямые+НР+СП): … ₽
  + непредвиденные 2%: … ₽
  + НДС 20%: … ₽
━━ ВСЕГО (общая цена с НДС): … ₽
⚙ Состав укрупнённый (типовой ИЖС) — отделка/проёмы/инженерка добавляются в шаблон.
Допущения: пятно квадратное P = 4·√(S/N); высота этажа H = 3.0 м; …
```
Геометрия двухэтажного дома 100 м²: S₁ = S/N = 50 м², a = √50, **P = 4·√50 ≈ 28.28 м**; объёмы
позиций — формулы над {S, N, P, H, S₁}, делятся на 100 (единица нормы — «100 м²/м³»). Допущения
выписываются честно в `assumptions`.

**Тяжёлое ТЗ по массе без ВОР** — fallback для стальных/бронзовых конструкций:

- расчёт по массе остаётся ориентиром `computed_assumed`, не защищаемой ЛСР;
- видимый ответ не показывает служебные `source_ref`/yaml/id ставок как доказательства;
- в ответ добавляются кандидаты ГЭСН из сборника 09 для ручной привязки к ВОР/Приложению 1;
- продолжения текущей сессии используют `retrieval_trace` предыдущего tool-расчёта: фраза
  «учти высотные работы» поднимает массу/ярусы из прошлого ответа и либо применяет явный
  коэффициент (`k=1,15`), либо честно просит высоту/ПОС/нормативное основание.

**Куда это идёт по плану (инверсия детерминизма).** Сейчас сметные каналы стоят keyword-**гейтом**
перед RAG. По решению от 2026-06-23 (см. `docs/AUDIT_DETERMINISM.md`) они становятся **инструментами
агент-роутера**: price/kac/stesn/lsr_assemble уже MCP-инструменты; `object_estimate` переходит с
шаблон-пресета на **RAG/LLM-рассуждение о составе работ** над корпусом норм + те же
инструменты-калькуляторы. Механизм счёта (код) не меняется — меняется кто его вызывает.

## 4. Источники базы (что откуда берётся)

| Что | Источник | Доступ | Импорт |
|---|---|---|---|
| **Нормы ГЭСН** (расход ресурсов) | ФГИС ЦС API `SearchEstimatedRates` | **бесплатно, без auth/квоты/гео** (префиксный поиск по шифру отдела) | `tools/gesn_bulk_import.py` (резюмируемый) → `data/gesn_base/gesn2022.parquet`, **42408 норм** |
| **PDF-корпус ГЭСН/ГЭСНм/ГЭСНр/ГЭСНп** | внешняя папка оператора | evidence/RAG-корпус, не замена parquet-базы расчёта | `POST /api/rag/index-external` by-reference; классификация `NORMATIVE/NTD_CONSTRUCTION` |
| **Цены ресурсов** | «Сплит-форма» ФГИС ЦС (файл, регион×квартал) | файл-выгрузка (большой, не в репо) | Инструменты → ФГИС ЦС / `POST /api/prices/import` → `data/price_base/*.parquet` |
| Цены **неучтённых** материалов | коммерческие предложения поставщиков | КАЦ ≥3 КП | [[ALGO-kac]] |

Принцип **локаль-first**: запрос сначала бьёт по локальной базе/книге; добор из сети — узкий
fallback с таймаутом ([[local-bases-untrusted-channel]]), сеть недоверенная по умолчанию.
ГЭСН-семя (`config/domain/gesn_seed.yaml`) содержит демо-норму эталона с полными тарифами и
**побеждает базу** при совпадении кода — эталон остаётся точным.

## 5. Принципы

- **ADR-11 — числа считает КОД, не LLM** ([[llm-minimalism-principle]]). Модель первична как
  диспетчер/связующий слой: понимает намерение, держит контекст, предлагает декомпозицию,
  находит недостающие слоты и объясняет результат. Харнесс первичен в расчёте: нормы, цены,
  коэффициенты, ВОР и итог проходят через инструменты/код; число без происхождения не результат.
  Если состав работ уже полный, модель не нужна как “сметчик-калькулятор” — она должна вызвать
  правильные инструменты, проверить применимость и показать blockers/MISSING. Геометрия, разворот
  норм, цены, НР/СП и хвост — детерминированная арифметика.
- **Числа цитируемы и проверяемы.** Каждая величина прослеживается до нормы/книги/Приказа; gold-тесты
  фиксируют эталон 11813.04 (`tests/test_gesn_service.py`, `tests/test_lsr_assembly_service.py`).
- **Знание ≠ веса.** Меняющиеся факты (ставки НР/СП, коэффициенты стеснённости, шаблоны объектов) —
  в редактируемых YAML (`config/domain/*`), а не в fine-tune ([[ALGO-smeta-ontology]]).
- НР/СП — из официальных **Приказов Минстроя 812/пр (НР) и 774/пр (СП)**, оба % от ФОТ, по виду
  работ (`config/domain/nr_sp.yaml`, поле `basis` несёт номер пункта Приказа).

## 6. Честные границы

- **Шаблоны объектной сметы — два: `wooden_house` (брус, типовой ИЖС) и `monolith_office`
  (офис/административка, монолит).** Запрос про **непокрытый** объект (напр. каркас/ангар) даёт
  **честный отказ**, а не выдумку: ловушка матча решена — пустой материал больше не лепит
  «деревянный дом» на любой объект (`wooden_house` требует сигнал «дерево»; «офис/здание» без
  материала уходит в `monolith_office`). Расширение — добавление шаблонов в
  `config/domain/object_templates.yaml` (данные, не логика).
- **Состав укрупнённый.** Деревянный дом — 7 позиций конструктива + теплоизоляция; отделка, окна/двери,
  инженерные сети — отдельные сборники, добавляются позициями в шаблон.
- **Базовые нормы из parquet несут материалы/машины, но не тариф ОЗП/ОТм** → ОЗП таких позиций = 0
  до загрузки тарифов; смета всё равно собирается на доступном (материалы + машины + НР/СП от ФОТ).
  Полная бухгалтерия (ОЗП≠0) — у норм с тарифом по разряду (как эталон 12-01-034-02 в семени).
- **Машинист-агрегат без разряда** → код не выводим → флаг «нет цены» (не молчаливый 0).
- **Геометрия грубая** (квадрат в плане, P = 4·√(S/N)); сложные обводы/проёмы — допущение, помечается.
- **Подбор кода нормы под произвольную работу ВОР** (наименование → ГЭСН) — отдельный шаг (ретрив по
  базе ГЭСН), пока не автоматизирован.
- **План** (см. `docs/AUDIT_DETERMINISM.md`): шаблон-пресет объектной сметы → **RAG-рассуждение о
  составе работ** над корпусом норм + инструменты-калькуляторы; детерминизм становится инструментами
  агент-роутера, RAG — основным механизмом.
