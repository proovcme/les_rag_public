# ALGO: Vision-извлечение и вопрос LoRA (Ц12/Ç13) — вердикт

Закрытие целей **Ц12** (harvest → бенч → решение по LoRA) и **Ц13** (бенч VL-модели). По
[[llm-minimalism-principle]] LoRA — последний инструмент: лестница «алгоритм → словарь → малая
модель → локальная LLM/VL → (только если не дотягивает) LoRA».

## Что доказано на реальных данных (чек-листы МФЗ Б4)

1. **OCR-мёртвые сканы — реальность, не теория.** 2/3 чек-листов (светильники/щиты) — мусор на всех
   страницах для Tesseract+rus. Это vision-кейс.
2. **Vision ЧИТАЕТ то, чего у OCR ноль — доказано с данными.** `qwen3-vl` (через ollama, `/api/chat`)
   разбирает таблицу щитов (АОРПН / Монтаж-расключение-ЛКИ / этаж / замечание) и **рукописную приёмку**
   («не тесн.», «удержано», подписи инспекторов + даты) на скане, где OCR = чистый мусор.
3. **Тулинг-развилка решена.** Корень пустых ответов — thinking-режим qwen3-vl не гасился в ollama
   0.30.10 (`think:false`/`/no_think` игнорировались). Рецепт: `RENDERER/PARSER qwen3-vl-instruct`
   (не `-thinking`) + prefill пустого `<think>\n\n</think>` последним assistant-сообщением → `done:stop`,
   чистый JSON. См. `deploy/ollama/qwen3-vl-nt.Modelfile`.
4. **Дешёвый рычаг бьёт молоток (3-й раз).** Описания полей в схеме (а не смена модели / не LoRA)
   починили маппинг: оборудование 0/15 → 15/15 на щитах; на розетках 17/17 строк со ВСЕМ рукописным —
   количество (3,2,1), отметка о приёмке («ГЛ 27.04 ГН 27.04»), позиция (артикулы 5602/5608).

## Вердикт по LoRA: НЕ нужна (для текущего корпуса)

Главный довод «за LoRA» — «рукопись теряется» — **снят**: база (vision + описания полей) рукопись
читает. Остаётся лишь символьная шумность почерка, которую LoRA *могла бы* подполировать, но данные
уже извлечены. Решение **гейтится числом** (ниже), и при текущих данных ожидаемо «не нужна».

- **LoRA только для ВОСПРИЯТИЯ (VL), не для знания** ([[cloud-providers-constraint]]): знание (ВОР/КАЦ/
  ЛСР) живёт в онтологии/RAG (правится, цитируется), не в весах.
- **Гейт:** делать узкую extraction-LoRA на Qwen3-VL ТОЛЬКО если field-accuracy локального VL+описаний
  не дотягивает на golden. eval-сет = он же train-данные (license-clean «lift»).

## Бенч (Ц13) — методология и статус

Харнесс: `tools/extract_bench.py` + `golden/extract_eval_set.json` (field-accuracy, как у datalab/lift).
Путь: скан → (OCR | vision) → `extract_service.aextract(schema)` → валидный JSON → сверка с ground truth.
- **Печатное** (OCR-чистое): OCR+4B+описания полей — header 100%, помещения ~71%/~100%-норм.
- **Рукопись/мусор**: qwen3-vl-nt + описания полей — извлекает (15/15, 17/17 выше).
- **Число field-accuracy на полном golden** считается командой:
  `uv run python tools/extract_bench.py --cases golden/extract_eval_set.json`
  (требует живой VL: ollama `qwen3-vl-nt`; на агенте/в CI без моделей — не прогоняется).

## Ограничения окружения (честно)

- Новые VL-модели сейчас НЕ подтянуть: Cloudflare R2 реестр ollama + HF-CDN режутся
  ([[cloudflare-blocked-use-openai-direct]]). Работаем тем, что уже скачано (qwen3-vl:4b/8b, gemma4:12b).
- Память: живой MLX-стек + VL не влезают в 24ГБ одновременно — VL-прогон требует освобождения GPU.
- Других сканов, кроме этих чек-листов, Олег не даёт → golden строится из них же.

## Итог

Ц12/Ç13 закрыты как **решение**: лестница держится (OCR+4B+описания для печатного → qwen3-vl-nt+описания
для рукописи/мусора → LoRA последней и **пока не нужна**). Инструменты (harvest_service, extract_bench,
extract_service, Modelfile) на месте; число field-accuracy — за командой выше при живом VL.
