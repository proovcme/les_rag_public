## Архитектурные ошибки (копим → финал-план)
1. **Гейты не аддитивны** — quality/CRAG/strict-retry/scope блокируют элементарный путь, компаундятся
   (см. AUDIT_RAG_ARCHITECTURE). Должны «метить, не зануляять».
2. **Перевёртыш ретрая** — «безопасная» попытка №2 строже первой → шанс ответить падает. (частично пофикшено)
3. **scope по имени датасета** — route_query целит в несуществующие имена (NTD_HVAC). Хрупко.
4. **Парс-планировщик не авто** — 398 файлов PENDING висят, не индексируются сами; health «degraded»
   из-за бэклога. Нужен авто-запуск планировщика ИЛИ health не должен деградировать на отложенном parse.
5. **Контракт payload чанка неединообразен** — `doc_name` пуст у части точек (источник в ответе тогда «»).
6. **Валидация входа чата неполная** — пустой вопрос → пустой ответ, а не graceful-сообщение/422.
### Защита памяти / MLX (агент)
7. **Двойной контур выгрузки** — MLXMemoryManager TTL + memory_guard_loop выгружают независимо (риск рассинхрона порогов).
8. **Нет жёсткого OOM-guard перед загрузкой** — защита только реактивная (по swap постфактум); холодная 14B на 24ГБ → hard-OOM.
9. **Нет warn при `MLX_VAL_MODEL==MLX_MODEL`** — оператор ловит холодные перезагрузки ([[mlx-model-choice-m4]]).
10. **Гонка check-then-unload** в TTL-loop (низкий риск).

### Почта-инжест (агент)
11. **Парс в основном потоке** — `/messages`/`/threads` делают полный re-parse всего ящика на КАЖДЫЙ GET, без кэша → узкое место.
12. **Нет дедупа писем по `Message-ID`** поперёк источников (IMAP + .olm → 2 документа в RAG).
13. **`.msg` тихо деградирует** без `extract_msg` (subject=имя файла, тело теряется); **`.olm` не извлекает вложения**.
14. **`thread_key` хрупок** — письма без Message-ID склеиваются по теме («Смета» + «Смета» = один тред).
15. **`/push` полу-молчаливая потеря** — `ok:True` даже если тело письма не залилось в RAG.

### Форматы файлов (агент)
16. **Парсинг блокирует event loop** — convert/OCR/subprocess синхронные; тяжёлый PDF/OCR вешает весь прокси («чат висит»).
17. **4 рассинхронных диспетчера расширений** — `converter.SUPPORTED` / `config.DEFAULT_RAG_UPLOAD_SUFFIXES` / `smart_index` / `document_router` расходятся; нет единого источника истины.
18. **Невалидный контракт результата** — `convert_to_markdown` отдаёт то None, то markdown, то sentinel-строку `[WARN]/[ERROR]` ВНУТРИ текста, то СЫРЫЕ БАЙТЫ (битый docx через markitdown) → мусор индексируется молча; потребитель не отличает успех от деградации.
19. **OCR синхронный без backpressure** — модель грузится/выгружается на каждый PDF («3x штраф»).
20. **Кодировки best-effort молча** — cp1251 → кракозябры без сигнала; нет chardet/флага качества.

## ПЛАН (финал) — по приоритету «живой, не хрупкий»

**П0 — фрагильность ядра РАГ (сделано в этом заходе):**
- ✅ гейты ответа аддитивны (sys_strict→синтез, concentrate 0.5→0.0); scope-промах→весь NTD-корпус. РАГ отвечает.
- ⏭ доделать: `evaluate_retrieval_quality`/CRAG/Т.О.С.К.А. — «метит, не зануляет» (ещё не валили кейсы, но потенциал).

**П1 — async-барьер парсинга/OCR (#16,#11,#19):** весь тяжёлый CPU/OCR/subprocess — в threadpool/очередь, НЕ в event loop. Единый барьер. (Корень «чат висит».)

**П2 — единый контракт (#17,#18,#5,#20):** один источник истины по форматам; `convert_to_markdown` → тип Result (ok/markdown/degraded+reason), НЕ sentinel-в-тексте и НЕ сырые байты; magic-bytes валидация; chardet для кодировок; единообразный payload чанка (doc_name всегда).

**П3 — self-healing индекс (#4):** авто-запуск парс-планировщика на pending (или health не degraded на отложенном parse). 398 файлов добить в корпус.

**П4 — OOM-guard + унификация выгрузки (#7,#8,#9):** жёсткий guard свободной памяти перед загрузкой; один контур выгрузки; warn при val==main.

**П5 — почта-устойчивость (#12,#13,#14,#15):** дедуп по Message-ID; .msg/.olm форматы + вложения; /push честный статус.

**П6 — мелочь:** валидация входа чата (#6, empty→graceful); ЮИ-аудит (не дошёл).
