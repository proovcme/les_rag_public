# LES3_PLAN — план работ ЛЕС 3 (формат для агентов)

**Обновлён:** 2026-06-10 · **Статус:** утверждённый черновик
**База:** аудиты ЛЕС 2 (RAG-ядро, CAD/BIM, АРТЕЛЬ, UI, LLM-контур) + спека PDF-препроцессора (2026-06-07) + ревизия технологий RAG (2026-06-10).

---

## Как пользоваться этим планом (агенту)

1. Перед любой задачей прочитай [AGENTS.md](../AGENTS.md) и [docs/CODE_MAP.md](CODE_MAP.md). Guardrails оттуда действуют всегда и важнее этого плана.
2. Бери **одну карточку задачи** (`W*.N`). В карточке: файлы, что сделать, критерий приёмки. Не расширяй скоуп.
3. Цикл: минимальный дифф → точечный тест → **`make verify`** → доменный гейт, если карточка помечена `[gate:domain]` (`uv run python tools/rag_golden_set.py --cases golden/domain_fire_hvac_set.json`, база 16/16).
4. Выполнив задачу — поменяй `[ ]` на `[x]` в этом файле и допиши однострочный итог (дата, коммит).
5. **Без явного одобрения оператора нельзя:** полный реиндекс (`[reindex]`-задачи), рестарты сервисов, `uv add` (`[dep]`-задачи), удаление данных, изменение дефолтов рантайма (`VALIDATOR_BACKEND` и т.п.).
6. Застрял или карточка противоречит коду — **остановись и доложи**, не угадывай.

Метки: `S/M/L` — сложность · `[reindex]` — нужно окно переиндексации · `[dep]` — новая зависимость · `[gate:domain]` — обязателен доменный гейт · `[live]` — нужны живые Qdrant/MLX для проверки.

## Цели версии

1. **Стабильность** — индексация любого объёма без таймаутов и присмотра; падение стадии не теряет работу.
2. **Качество** — эмбеддер видит весь текст чанка (без молчаливых обрезок); ответы на структурные вопросы (пункты ГОСТ, этажи/системы BIM, сканы).
3. **Мультиплатформа** — Win/Mac первого класса, Linux через docker.
4. **Провайдер-агностик** — MLX, Ollama, llama.cpp, LM Studio, любой OpenAI-совместимый эндпоинт.
5. **Установка одной командой.**

## Зафиксированные архитектурные решения (ADR)

| # | Решение | Обоснование |
|---|---|---|
| ADR-1 | Эволюция, не переписывание: proxy/Qdrant/SQLite-метабаза/шеллы остаются | Аудит показал зрелое ядро; узкие места — в слоях |
| ADR-2 | Протокол OpenAI API — шина для всех LLM-провайдеров | mlx_host уже говорит `/v1/*`; Ollama/llama.cpp/LM Studio совместимы |
| ADR-3 | **Реранкер — cross-encoder модель, НЕ LLM-промпт.** Существующий `backend/reranker.py` (Qwen3-4B, «оцени 0-10») заменяется, не дорабатывается | Ревизия 2026-06: LLM-pointwise реранк устарел; cross-encoder быстрее на порядок, без парсинга, без конкуренции за Metal |
| ADR-4 | Лексическая ветка — learned sparse (BGE-M3 sparse / SPLADE) в Qdrant; самописный стеммер не развиваем | BGE-M3 уже в профилях, используется только dense-выход; Qdrant нативно держит sparse |
| ADR-5 | Конвертация PDF — layout-aware парсер (Docling или MinerU), pymupdf4llm остаётся fallback | Кросс-страничные таблицы спецификаций — текущая боль |
| ADR-6 | Сканы — отдельный визуальный индекс (ColPali/ColQwen, мультивекторы Qdrant), OCR для сканов не основной путь | Консенсус 2026: на сканах визуальный ретрив бьёт OCR-пайплайн |
| ADR-7 | Чанк-размеры в **токенах**, жёсткая проверка ≤ seq_len эмбеддера при старте | Сейчас qwen-профиль 1550 символов > 512 токенов — хвосты молча теряются |
| ADR-8 | Отвергнуто/отложено: полный GraphRAG (дорог; LightRAG — только эксперимент W6.6), LLM-контекстуализация чанков (есть механическая версия), семантический чанкер по эмбеддингам (structure-aware под ГОСТ лучше) | Не тащить хайп без выигрыша на наших golden sets |
| ADR-9 | **Оркестр облако+локал**: маршрутизация задач между локальными и облачными моделями по матрице «класс задачи × чувствительность данных × доступность». **Облако — только OpenRouter и OpenAI API** (решение оператора, 2026-06-10); прямых интеграций других вендоров не делаем. Оба говорят на OpenAI-протоколе → облако ездит по той же шине ADR-2, что и локальные провайдеры — отдельного SDK не нужно. OpenRouter — гибкость (десятки моделей через один эндпоинт, A/B без кода), OpenAI напрямую — Batch API (−50%) и промпт-кэш. Чувствительность — флаг на датасете: `P0 local-only` (почта, договоры, персональные данные) / `P1 cloud-ok` (нормативка, открытые справочники) / `P2 cloud-ok-с-согласия` (чертежи и данные проектов). Облако недоступно → деградация на локальный fallback, не отказ | Локальные модели закрывают 80% потока дёшево и приватно; облако даёт качество там, где локальные физически не тянут (рукопись, генерация семейств, сложный анализ чертежей); один протокол — меньше кода |
| ADR-10 | **Рукописные пометки — vision-VLM облачного класса + human-in-the-loop**: распознавание рукописи на чертежах локальными моделями ненадёжно; первичный путь — облачная vision-модель (GPT-5.5 через OpenAI/OpenRouter) со structured output, каждое извлечённое значение подтверждается человеком до записи в журнал объёмов. Локальная Gemma 4 12B (мультимодальная) — ступень предклассификации кропов до облака | Цена ошибки в учёте объёмов — деньги; рукопись — признанно сложный кейс даже для облачных VLM |
| ADR-11 | **LLM-минимализм**: LLM — последний инструмент, не первый. Порядок выбора для любой подзадачи: (1) детерминированный алгоритм/SQL/regex → (2) словарь/справочник/кэш → (3) специализированная малая модель (cross-encoder, эмбеддер, классификатор) → (4) локальная LLM → (5) облачная LLM. Каждый LLM-вызов в коде должен отвечать на вопрос «почему это нельзя сделать алгоритмом» — комментарием или ссылкой на ADR. Числа, суммы, агрегации LLM **никогда не считает** — только форматирует посчитанное. Роль LLM в системе: **понимание, объяснение, мост между человеком и системой**. Канон: статья оператора [«Границы применения LLM»](https://ai.ovc.me/llm-boundaries) — 7 запретов (UI, расчёты, хранение, XML, проверка норм, визуальное сравнение, инженерные решения) и целевая архитектура «алгоритм находит факты — LLM переводит на человеческий» | Скорость, детерминизм, тестируемость, цена. Лучшие части ЛЕС 2 (clause lookup, rules-валидатор, KOT-роутер) — уже алгоритмы; это направление, а не исключение |

**Что НЕ устарело и НЕ трогать без причины:** Qdrant; structure-aware сплиттер (ГОСТ-пункты); детерминированный clause lookup; C-RAG валидация как концепция; semantic cache; golden-гейты; логика памяти `backend/mlx_adapter.py`.

**Watch-list (наблюдаем, не делаем):** [apple/coreai-models](https://github.com/apple/coreai-models) — Core AI (`.aimodel`, требует **macOS 27+**; прод-Mac на 26 — блокер). Единственная интересная точка входа: замена Core ML-эмбеддера (наша боль — fixed-shape .mlpackage, батч 1, процесс-изоляция; в каталоге есть Qwen3). Критерий пересмотра: обновление прод-Mac до macOS 27 + динамические батчи в Core AI. Бонус-кандидаты на потом: SAM/CLIP для детекции зон пометок (W8.2), Whisper для голосовых заметок. LLM-инференс не трогаем — MLX остаётся (ADR-1).

## Рекомендации по моделям (срез июнь 2026, пересматривать ежеквартально)

### Локальный ярус (Mac / MLX; на Windows — те же через Ollama)

| Роль | Модель | Замечания |
|---|---|---|
| main (генерация) | **Qwen3.6-35B-A3B** (MoE, 3.5B активных) | Кандидат на замену текущей Qwen3-14B: быстрее на токен при выше качестве; 262K контекста. На 64GB+ RAM — вариант Qwen3.5-122B-A10B. Принимать через golden set, не по бенчмаркам |
| validator | Qwen3-4B (текущая) или 8B-класс | Каскад rules→LLM (W3.3) важнее размера модели |
| embedder | Qwen3-Embedding-0.6B (текущая) | Не менять без явной причины — реиндекс дорог; проверить более свежие только вместе с окном реиндекса W2 |
| reranker | Qwen3-Reranker-0.6B / bge-reranker-v2-m3 | См. W2.2 (ADR-3) |
| vision/мультимодал локальный | **Gemma 4 12B** (Ollama `gemma4:12b`; установлена 2026-06-11, Ollama 0.30.7) | Мультимодальная (текст+изображение+аудио), reasoning, 7.6 ГБ. **Полевой вывод 2026-06-11:** на прод-Mac (24 ГБ) рядом с MLX-стеком выталкивает систему в swap (86% → admission блокирует чат) и непрактично медленна на RAG-контекстах интерактивно. Роли: **только офлайн-окна** — предклассификация фото/кропов (W8.2), пакетные задачи при выгруженном MLX. Интерактивный чат — нет. Восстановление после: `ollama stop gemma4:12b` + рестарт MLX + дождаться дренажа swap |
| OCR локальный | GLM-OCR (Mac) — сравнить с Gemma 4 12B | Только печатный текст; рукопись — в облако (ADR-10) |

### Облачный ярус (только OpenRouter + OpenAI — ADR-9; цены за 1M токенов вход/выход)

| Роль | Модель | Цена | Когда |
|---|---|---|---|
| Тяжёлый анализ, vision чертежей/рукописи, генерация семейств | **GPT-5.5** (OpenAI или через OpenRouter) | $5/$30 | Задачи, где локальные не тянут; reasoning + vision + structured outputs; контекст 1M. **Внимание: >272K входных токенов — наценка 2×/1.5× на всю сессию, в т.ч. в Batch** — длинные документы резать до порога |
| Рабочая лошадка облачных задач | GPT-5.4 | $2.50/$15 | Массовая классификация пометок, нормоконтроль-проверки — лучший баланс цена/качество |
| Дешёвая классификация/экстракция | GPT-4.1 Nano / mini-класс через OpenRouter | от $0.10 вход | Маршрутизация, простая экстракция полей |
| A/B и страховка от вендор-лока | любая модель через OpenRouter | по прайсу модели | Один эндпоинт — сравнение моделей на наших golden-наборах без изменения кода |

Снижение стоимости: **OpenAI Batch API −50%** на всё неинтерактивное (ночные пакеты фото, массовый нормоконтроль) — только при прямом OpenAI, у OpenRouter батчей нет; **промпт-кэш OpenAI −75-90%** на повторных префиксах (системные промпты, справочник ФОП у генератора семейств) — кэш автоматический, но требует стабильного префикса промпта. Structured outputs — обязательны для всех экстракционных задач: убирают парсинг свободного текста.

### Матрица маршрутизации (политика диспетчера, W3.5)

| Класс задачи | Данные P0 | Данные P1/P2 |

## Приложение А — правки к спеке PDF-препроцессора (обязательны для W1.3)

Спека 2026-06-07 принята. Четыре правки:

1. **Оригиналы не удалять**: перемещение в `_originals/` (флаг `--archive-dir`); удаление — только явным `--delete-originals`. Причина: guardrail «беречь RAG_Content».
2. **State-файл идемпотентности** (`.pdf_preprocess_state.json`: path→mtime,size,hash): очистка меняет mtime/size, метабаза по ним отслеживает документы ([qdrant_adapter.py:386](../backend/qdrant_adapter.py)) — без state повторный прогон переиндексирует уже готовые файлы.
3. **Сплит по структуре**: жадная упаковка страниц по фактическим байтам + предпочтение границ закладок/секций; в payload частей `part_index/part_total/original_name` (источники в чате склеиваются, clause lookup находит семейство по `doc_name LIKE`).
4. **Параллельно лечить причину**: per-file таймаут с постраничной батч-конвертацией PDF — в W1.4; препроцессор — оптимизация, не обязательный костыль.

Остальное (CLI, коды выхода 0/1/2, atomic tmp→rename, JSON-лог, тест-план ~13 тестов, имена `_частьN.pdf`) — как в спеке.
