# Поиск полезных данных на строительных сканах

> Как ЛЕС достаёт данные из сканов исполнительной документации, **понимая тип**
> таблицы (ведомость / экспликация / журнал / спецификация / чек-лист), и зачем
> это устроено именно так.

---

## 1. Зачем

На стройке сотни сканов: чек-листы приёмки, ведомости материалов и объёмов работ,
экспликации помещений, журналы, спецификации, кабельные журналы — часто рукописные,
повёрнутые, «вшитые» в углы больших чертежей. Руками это не разобрать.

Задача ЛЕСа — **извлечь полезные данные**, а «полезное» зависит от **типа**:
ведомость материалов → объёмы/закупки, экспликация → реестр помещений, журнал →
хронология, спецификация → комплектация, чек-лист → факт приёмки. Поэтому машина
обязана не просто «прочитать таблицу», а **понять, что это за таблица**, и положить
данные в правильное место (`route`).

Принципиальная установка: распознавание — не цель, а вход. Цель — **зафиксировать
принятый объём / достоверные данные**, где оператор подтверждает или правит, а не
доверяет модели слепо. Поэтому verify-режим выдаёт артефакт для сверки, а не
«ответ».

---

## 2. Принцип: LLM-минимализм (ADR-11)

Каждый шаг решается самым дешёвым достаточным инструментом; LLM — последним.

| шаг | инструмент | почему не LLM |
|---|---|---|
| найти таблицы на листе | **CV** (линии сетки, OpenCV) | детерминировано, мгновенно |
| тип таблицы | **сигнатура шапки** (словарь) | объяснимо, бесплатно |
| тип, если шапки мало | дешёвый **vision на одно название** | один короткий вызов |
| содержимое таблицы | **vision** (qwen3-vl) | только тут нужен «глаз» |
| принятый объём (рукопись) | **оператор** | надёжнее галлюцинаций |

Vision дорог и серийн (одна GPU). Всё, что можно снять алгоритмом/словарём, снимаем
без него.

## 3. Как это работает — конвейер

```
скан (PDF/изображение)
  │
  ├─[1] детект таблиц (CV)         table_detect.detect_table_regions → регионы [x0,y0,x1,y1]
  │
  ├─[2] по каждому региону:
  │      ├─ извлечение (vision)    verify_service._vision_extract_rows → строки JSON
  │      └─ классификация типа     doc_classifier.classify_table → {тип, route, conf}
  │
  └─[3] фиксация
         ├─ оператор сверяет/правит (артефакт verify, колонка «Принято»)
         └─ или пакетный отчёт      scan_batch → batch_report.json
```

### [1] Детект таблиц — `proxy/services/table_detect.py`
Морфология **горизонтальных + вертикальных линий** (вытянутые ядра erode/dilate) →
маска сетки → блоки → bounding-box каждой таблицы (нормированные `0..1`).
- **Фрейм-фильтр**: рамка листа-чертежа во весь лист отбрасывается, если есть
  таблицы поменьше; одиночная страница-таблица (чек-лист) — остаётся.
- **Дедуп**: вложенные регионы выкидываются.
- Без LLM. Заменяет ручную рамку → разблокирует автономный батч.
- Тюнинг: `TABLE_DETECT_MIN_AREA` (доля площади листа).

### [2a] Извлечение — `verify_service._vision_extract_rows / _vision_call`
Локальный vision `qwen3-vl-nt` (ollama, не-thinking рендерер + prefill пустого
`<think>` → чистый JSON). Качество тянется **промптом**, а не сменой модели:
- **доменный глоссарий** (`VERIFY_VL_GLOSSARY`) — графы приёмки, марки щитов
  ШР/ШО/ШС/РШТ; различение похожих кириллических букв (А/Л, П/И, О/С) → читает
  «АОРПИ», «КОРПУС» вместо «ЛОРПИ», «КОРИУС»;
- **дизамбигуация цифр** — «5» это пять, не латинская «S» (было: ШС6**S** →
  стало ШС6**5**); точки в марках дословно;
- **анти-галлюцинация** — «пиши только что видишь; нечитаемо → `[]`»; модель
  иначе выдумывала generic-смету на нечитаемом скане;
- **тайлинг высоких** (`VERIFY_VL_TILE_ASPECT`) — узкая таблица на весь лист
  режется на горизонтальные полосы с перекрытием;
- **авто-поворот** (`VERIFY_VL_AUTOROTATE`) — таблицы на больших листах часто
  повёрнуты на 90°; если прямой проход пуст, пробуем CW/CCW;
- **кроп региона** (`_region_image`) — рендер под-прямоугольника в высоком
  разрешении (fitz clip ×`VERIFY_REGION_SCALE`); координаты выделения совпадают с
  показанным рендером (одна ориентация по `pg.rect`) → мапятся 1:1.

### [2b] Классификация типа — `proxy/services/doc_classifier.py`
`classify_table(columns, rows, title?)`:
1. **Сигнатура шапки** — для каждого типа набор ключевых граф (вес `1/N`); название
   таблицы (если известно) — вес `1.5`. Порог `0.34`, иначе «неизвестно».
2. **title-фолбэк** — если шапки мало (экспликация = «номер/Имя», conf 0.29),
   дешёвый `_read_title_vision` читает **только название** → реклассификация
   (`VERIFY_CLASSIFY_TITLE=0` отключает).

Типы → `route`: ведомость_материалов→`объёмы_закупки`, ВОР→`объёмы_работ`,
экспликация→`реестр_помещений`, спецификация→`комплектация`, журнал→`хронология`,
кабельный_журнал→`кабели`, чек-лист_приёмки→`приёмка`, основная_надпись→`метаданные`.

### [3] Фиксация — артефакт verify + пакетный отчёт
**Артефакт** (`sovushka/pages/verify.py`) — не вкладка, а артефакт чата (как
визуализатор Клода): сверху скан (`ui.interactive_image`, тащишь рамку мышью →
«⤵ Извлечь выделенное»), снизу таблица (правятся шапка и ячейки). Сверху — **бейдж
типа**. Если есть графа количества — добавляется зелёная колонка **«Принято
(объём)»**, преднаполненная печатным Кол-во: оператор правит только отличия
(рукописные исправления приёмки 154→132), галочка ✓ = принято как есть. Сохранение
(`/api/verify/save`) = **принятая выписка + ground truth для бенча**.

**Пакет** (`tools/scan_batch.py`): папка → по листу детект → извлечение по регионам
+ классификация → `batch_report.json` + сводка по типам.

## 4. Три режима использования

| режим | как | когда |
|---|---|---|
| **один файл в чате** | `проверь объёмы <путь>` → артефакт | обычная таблица/чек-лист |
| **большой лист** | артефакт → выдели таблицу рамкой → ⤵ | таблица внутри чертежа |
| **пакет** | `scan_batch.py <папка>` | папка сканов, автономно |

Большой лист (> `VERIFY_AUTO_EXTRACT_MAX_MP`, 12 Мп) без региона **не молотится**
целиком (это было ~3 мин каши) — сразу «выдели таблицу рамкой» (1.8с). В пакете
рамку заменяет CV-детект.

Пакет — два подрежима: `--classify-only` (разведка «что в пачке», 11 листов за 6с,
без vision) и полное извлечение (фоном).

## 5. Доказательство

Лист `КЛ Д … приняты с объемами` (чертёж 9742×3521, /Rotate=90, таблица в углу):
автономный батч нашёл **4 таблицы** и назвал каждую — Журнал (6), Экспликация (38),
**Ведомость материалов (10)**, Экспликация (4). Ведомость материалов — кабель/гофра
с количествами **точь-в-точь**: 154, 696, 274, 330, 146, 556, 197, 427, 235, 95.
Ноль кликов оператора.

Бенч (`tools/verify_bench.py`, эталон = подтверждение оператора): на чек-листе
щитов шапка 5/6 (83%, АОРПИ/КОРПУС верно), объёмы recall 0→50% после фикса «5≠S».

## 6. Ключевые решения и находки

- **«ИД … с объемами» ≠ таблица.** Часть таких сканов — план этажа (расстановка),
  а не ведомость; vision корректно вернёт `[]` (анти-галлюцинация). Реальный ВОР
  («Ведомость материалов») сидит в углу того же чертежа.
- **Поворот — частая причина мусора.** Большие листы ландшафтные с /Rotate; чтение
  велось «боком». Показ и кроп держим в одной ориентации (по `pg.rect`).
- **Рукопись — человеку.** Vision не читает надёжно рукописные пометки приёмки
  (повторял число на строках с галочками). Поэтому «Принято» правит оператор —
  это и есть акт приёмки, надёжнее OCR.
- **Глоссарий не навязывает графы.** Пример «Поз./Наименование оборудования»
  затекал в шапку экспликации (реально «номер/Имя») — теперь «возможные, НЕ
  обязательные; читай реальную шапку».

---

## 7. Ограничения и что дальше

- **Скорость**: vision серийный (1 GPU) ≈ 2.4 мин/таблица → полная папка из 57 с
  извлечением = часы. Разведка мгновенна; извлечение — фоном/ночью или подвыборкой.
- **Классификатор** иногда путает (кабельную таблицу с датами назвал «Журнал») —
  чинится сигнатурами/строже title-фолбэком.
- **Мультистраничные PDF** — пока берём стр.0.
- Дальше: «тип без полного извлечения» (детект + мелкий title-кроп → дёшево),
  точность классификатора, сводный отчёт по всей папке.
